Google ADK Part 2 — Multi-Agent Systems with Sub-Agents

← Back to Home

In Part 1 we built a single agent with a custom tool and ran it via adk run and adk web. That works great for focused tasks — but real-world assistants need to handle many kinds of requests. Instead of cramming every capability into one agent, ADK lets you split responsibilities across specialised sub-agents and wire them together under a single orchestrator agent.

This post covers exactly that pattern:

  • What sub-agents are and why they matter
  • Building two specialised sub-agents, each with their own tool
  • Building an orchestrator that has its own tool and delegates to sub-agents
  • How ADK routes requests automatically — no manual if/else logic
  • Running and inspecting the system with adk web

1. The Multi-Agent Idea

In ADK the same Agent class you used in Part 1 can act as an orchestrator. You simply pass a list of other Agent instances to its sub_agents parameter. When a message arrives the orchestrator — powered by Gemini — reads each sub-agent's description and decides whether to answer itself or hand the request off to a specialist.

Think of it like a front-desk manager who handles general questions personally but forwards specialised enquiries to the right department. The routing decision is made by the model; you only define who each agent is and what they do.

The system we'll build looks like this:

root_agent  (orchestrator)
├── tool: get_current_time()        # answers directly
├── sub_agent: weather_agent
│   └── tool: get_weather(city)
└── sub_agent: calculator_agent
    └── tool: calculate(expression)

A single user-facing agent backed by two specialised sub-agents. The orchestrator can answer time queries itself and delegates weather or maths queries to the appropriate specialist.

2. Project Structure

We keep one folder per agent project. All agents live inside the same folder so ADK can find them: 

my_project/
├── .env                        # GOOGLE_API_KEY
└── travel_assistant/
    ├── __init__.py
    └── agent.py                # all agents defined here

You can split agents across multiple files as your project grows, but for clarity we'll keep everything in one agent.py here.

3. Building the Sub-Agents

Sub-Agent 1: Weather Agent

This agent is responsible for one thing only — answering weather questions. It has a get_weather tool and an instruction that keeps it focused on that domain. 

from google.adk.agents import Agent


def get_weather(city: str) -> dict:
    """
    Returns the current weather for a given city.

    Args:
        city: The name of the city.

    Returns:
        A dict with temperature, condition, and humidity.
    """
    mock_data = {
        "london":   {"temperature": "14°C", "condition": "Cloudy",  "humidity": "78%"},
        "new york": {"temperature": "22°C", "condition": "Sunny",   "humidity": "55%"},
        "tokyo":    {"temperature": "18°C", "condition": "Rainy",   "humidity": "85%"},
        "paris":    {"temperature": "16°C", "condition": "Windy",   "humidity": "70%"},
    }
    key = city.lower()
    if key in mock_data:
        return {"city": city, **mock_data[key]}
    return {"city": city, "error": f"No data available for '{city}'."}


weather_agent = Agent(
    name="weather_agent",
    model="gemini-2.0-flash",
    description=(
        "Handles all weather-related questions. "
        "Use this agent when the user asks about temperature, forecast, or current conditions in any city."
    ),
    instruction=(
        "You are a weather specialist. "
        "Always use the get_weather tool to look up conditions before answering. "
        "Present results clearly and conversationally."
    ),
    tools=[get_weather],
)

The description is what the orchestrator reads to decide whether to delegate here. Keep it precise — it directly affects routing accuracy.

Sub-Agent 2: Calculator Agent

This agent handles arithmetic. It has a calculate tool that safely evaluates a mathematical expression string and returns the result. 

def calculate(expression: str) -> dict:
    """
    Evaluates a mathematical expression and returns the result.

    Args:
        expression: A string containing a valid mathematical expression,
                    e.g. '(12 + 8) * 3 / 4'.

    Returns:
        A dict with the original expression and its numeric result,
        or an error message if the expression is invalid.
    """
    try:
        # ast.literal_eval cannot evaluate expressions; use eval with a
        # restricted namespace to prevent arbitrary code execution.
        allowed = {"__builtins__": {}}
        result = eval(expression, allowed)  # noqa: S307
        return {"expression": expression, "result": result}
    except Exception as e:
        return {"expression": expression, "error": str(e)}


calculator_agent = Agent(
    name="calculator_agent",
    model="gemini-2.0-flash",
    description=(
        "Handles all mathematical calculations and arithmetic questions. "
        "Use this agent when the user wants to compute, calculate, or evaluate a numeric expression."
    ),
    instruction=(
        "You are a calculation specialist. "
        "Always use the calculate tool to evaluate expressions. "
        "Show the expression and the result clearly."
    ),
    tools=[calculate],
)

4. Building the Orchestrator

The orchestrator is also just an Agent. Two things make it different from the sub-agents:

  • It lists the sub-agents in sub_agents.
  • Its instruction tells it when to handle a request itself versus when to delegate.

The orchestrator also has its own direct tool — get_current_time — that it handles personally without delegating anywhere. 

from datetime import datetime, timezone


def get_current_time() -> dict:
    """
    Returns the current UTC date and time.

    Returns:
        A dict with the current UTC datetime as a string.
    """
    now = datetime.now(timezone.utc)
    return {"utc_time": now.strftime("%Y-%m-%d %H:%M:%S UTC")}


root_agent = Agent(
    name="travel_assistant",
    model="gemini-2.0-flash",
    description="A general-purpose travel assistant that handles weather, calculations, and time queries.",
    instruction=(
        "You are a helpful travel assistant. "
        "For weather questions, delegate to the weather_agent. "
        "For calculations or arithmetic, delegate to the calculator_agent. "
        "For current time or date questions, use the get_current_time tool directly. "
        "For anything else, answer from your own knowledge. "
        "Always be concise and friendly."
    ),
    tools=[get_current_time],
    sub_agents=[weather_agent, calculator_agent],
)

5. How Delegation Works

There is no routing code — ADK handles it through the model. Here is what happens under the hood when a message arrives at root_agent:

  1. Gemini receives the user message alongside the orchestrator's instruction.
  2. ADK automatically appends the description of each sub-agent to the context, so Gemini knows what specialists are available.
  3. Gemini decides: answer with its own tool, delegate to a sub-agent, or answer directly from knowledge.
  4. If it delegates, ADK transfers the conversation to that sub-agent. The sub-agent runs with its own instruction and tools and returns a response.
  5. The final response is delivered back to the user from root_agent.

This means the description field is critical. If two sub-agents have vague or overlapping descriptions, Gemini may route incorrectly. Be specific about what each agent handles — treat the description like a job posting.

A practical tip: start the description with the trigger condition:

# Too vague — Gemini may not delegate reliably
description="A helpful agent."

# Good — clear trigger condition
description=(
    "Handles all weather-related questions. "
    "Use this agent when the user asks about temperature, forecast, or conditions in any city."
)

6. Running the Multi-Agent System

The commands are identical to Part 1 — ADK doesn't care how many agents are in the folder:

# Terminal chat
adk run travel_assistant

# Web UI
adk web
Example session with adk run
User: What time is it?

Agent: The current UTC time is 2026-03-27 09:14:32 UTC.

User: What's the weather in Paris?

Agent: The weather in Paris right now is windy with a temperature of 16°C
       and humidity at 70%. Might want to bring a light jacket!

User: What is (450 / 9) * 7?

Agent: (450 / 9) * 7 = 350.0

Each of those three messages was handled differently: the time question was answered by the orchestrator's own tool, the weather question was delegated to weather_agent, and the maths question went to calculator_agent — all transparently from the user's perspective.

Inspecting delegation in adk web

Open http://localhost:8000 after running adk web. The event timeline on the right side of the chat will show:

  • Which agent handled each turn (orchestrator or a sub-agent)
  • The tool call made, the arguments passed, and the value returned
  • The full delegation chain — useful when debugging unexpected routing

If you see the orchestrator handling a request that should have gone to a sub-agent, the event timeline is the fastest way to diagnose it: check what description the model was working with and refine it.

7. A Few Practical Guidelines

Give each agent one clear responsibility

Sub-agents work best when they have a tight, well-defined scope. An agent that handles "weather and news and stocks" is harder for the orchestrator to route to accurately than three separate agents with clear domains.

Sub-agents inherit session context

When the orchestrator delegates to a sub-agent, the full conversation history travels with it. The sub-agent sees what the user said and what the orchestrator replied — it's not starting from scratch. This means sub-agents can refer to earlier context without needing to re-ask the user.

Orchestrator instruction vs sub-agent instruction

These serve different purposes. The orchestrator's instruction tells it how to route and coordinate. Each sub-agent's instruction tells it how to handle and respond. Keep them focused on their respective roles — avoid repeating the same guidance in both.

Number of sub-agents

There is no hard limit, but in practice routing accuracy starts degrading as the number of sub-agents grows and descriptions become similar. For a handful of distinct capabilities (3–6 agents) the pattern works extremely well out of the box.

8. Conclusion

The multi-agent pattern in ADK is deliberately simple: every agent is the same Agent class, sub-agents are just entries in a list, and the routing logic lives entirely in natural-language descriptions rather than code. That makes the system easy to extend — adding a new capability is as simple as writing a new sub-agent and appending it to sub_agents.

  • Sub-agents are regular agents — same class, just nested inside an orchestrator.
  • Routing is description-driven — Gemini reads description to decide where to delegate.
  • Orchestrators can have their own tools — they don't have to delegate everything.
  • adk web shows the full delegation chain — your best debugging tool.

The next step beyond this is workflow agents — SequentialAgent and ParallelAgent — which give you explicit control over execution order when tasks need to run in sequence or in parallel. That's a topic for the next post.

One agent per responsibility, one clear description per agent — that's the key to a multi-agent system that actually works.
