Most tutorials show you how to build an AI agent. Very few show you how to make it remember. In this series we follow a single customer support chatbot as it evolves from a completely forgetful agent into a system with deep, persistent, semantically aware memory. Every part adds one layer of capability. By the end you will understand why the memory architecture you choose directly impacts your users' experience and your infrastructure bill.

Part 1 starts at the beginning: a simple SQLite database to store conversation history. We cover both Google ADK and LangChain implementations side by side, then show you why sync I/O becomes a bottleneck and how to fix it with async.

1. Why Agents Forget

Imagine calling a support line and the agent picks up. You explain your problem. They put you on hold. They come back - and have completely forgotten everything you just said. You explain again. Hold. Forgotten again. This is exactly what happens when you build an AI agent without memory.

The reason is structural. Large language models are stateless: each API call is independent. The model receives a block of text, produces a response, and discards everything. There is no persistent internal state between calls. When your agent calls the LLM for turn 2, it has no built-in recollection of turn 1 unless you explicitly send turn 1 back in the request.

This is not a bug; it is a deliberate design choice that makes LLMs scalable and easy to deploy. But it means memory is your responsibility as the developer. You must decide what to store, where to store it, and how to retrieve it.

What "memory" means for a support bot: The bot should recall what problem the user described, what solutions were suggested, whether an order number or ticket ID was mentioned, and what outcome was reached - within the same session and ideally across sessions too.

2. What We Are Building

Our running example throughout this series is a customer support chatbot for an e-commerce platform. The bot handles order queries, returns, and general account questions.

In Part 1 the bot can only remember the current conversation. If you close the tab and open a new one, it forgets you entirely. That is the honest starting point. Parts 2 and 3 will change this significantly.

3. SQLite 101 in Plain English

SQLite is a database that lives in a single file on disk. There is no server process, no network connection, no configuration. You just point your code at a .db file and start reading and writing.

Think of it as a very organised spreadsheet. A table is like a sheet. A row is one record (one message, one session). A column is a field within that record (the text, the timestamp, the role). SQL lets you ask questions like “give me all rows in the messages table that belong to session X, sorted by time.”

SQLite is excellent for development and low-to-moderate traffic production apps. It is already installed with Python; no pip install needed for the basic version.

Why Not Just Use a Python List?

You could store conversation history in a Python list in memory. Many tutorials do exactly this. But lists die when your process restarts. They cannot be shared across multiple worker processes. They cannot be queried efficiently. And they have no TTL; they just grow forever. A database gives you durability, shareability, queryability, and control.

4. Designing the Schema

We need two tables. The first tracks sessions (each unique conversation) and the second stores the individual messages within those sessions. 

-- sessions: one row per conversation
CREATE TABLE IF NOT EXISTS sessions (
    session_id  TEXT PRIMARY KEY,   -- e.g. "user_42_2026-05-19"
    user_id     TEXT NOT NULL,      -- the customer's identifier
    created_at  REAL DEFAULT (unixepoch())
);

-- messages: one row per turn (user or model)
CREATE TABLE IF NOT EXISTS messages (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id  TEXT NOT NULL,
    role        TEXT NOT NULL,      -- "user" or "model"
    content     TEXT NOT NULL,
    ts          REAL DEFAULT (unixepoch()),
    FOREIGN KEY (session_id) REFERENCES sessions(session_id)
);

The FOREIGN KEY link means every message must belong to a known session; the database enforces this for you. The ts timestamp on every message lets you retrieve history in chronological order, which matters a great deal when replaying context to the LLM.

5. Sync Implementation

Google ADK + SQLite

ADK's Runner manages sessions internally, but we can persist history ourselves by saving messages after each turn and loading them at session start. We use ADK's InMemorySessionService for in-flight session management and write to SQLite for durability. 

import sqlite3
from google.adk.agents import Agent
from google.adk.runners import Runner
from google.adk.sessions import InMemorySessionService
from google.adk.agents.callback_context import CallbackContext

DB_PATH = "support.db"

# --- Database helpers ---

def init_db():
    conn = sqlite3.connect(DB_PATH)
    conn.execute("""
        CREATE TABLE IF NOT EXISTS sessions (
            session_id TEXT PRIMARY KEY,
            user_id    TEXT NOT NULL,
            created_at REAL DEFAULT (unixepoch())
        )""")
    conn.execute("""
        CREATE TABLE IF NOT EXISTS messages (
            id         INTEGER PRIMARY KEY AUTOINCREMENT,
            session_id TEXT NOT NULL,
            role       TEXT NOT NULL,
            content    TEXT NOT NULL,
            ts         REAL DEFAULT (unixepoch()),
            FOREIGN KEY (session_id) REFERENCES sessions(session_id)
        )""")
    conn.commit()
    conn.close()

def load_history(session_id: str) -> list[dict]:
    conn = sqlite3.connect(DB_PATH)
    rows = conn.execute(
        "SELECT role, content FROM messages WHERE session_id=? ORDER BY ts",
        (session_id,)
    ).fetchall()
    conn.close()
    return [{"role": r, "content": c} for r, c in rows]

def save_message(session_id: str, role: str, content: str):
    conn = sqlite3.connect(DB_PATH)
    conn.execute(
        "INSERT INTO messages (session_id, role, content) VALUES (?,?,?)",
        (session_id, role, content)
    )
    conn.commit()
    conn.close()

# --- ADK callbacks to hook into the turn lifecycle ---

def before_agent(callback_context: CallbackContext):
    """Load history from SQLite and inject it into session state."""
    session_id = callback_context.state.get("session_id", "default")
    history = load_history(session_id)
    callback_context.state["history"] = history

def after_agent(callback_context: CallbackContext):
    """Persist the latest turn to SQLite after the agent responds."""
    session_id = callback_context.state.get("session_id", "default")
    last_user = callback_context.state.get("last_user_message", "")
    last_model = callback_context.state.get("last_model_response", "")
    if last_user:
        save_message(session_id, "user", last_user)
    if last_model:
        save_message(session_id, "model", last_model)

# --- Agent definition ---

agent = Agent(
    name="support_bot",
    model="gemini-2.5-flash",
    instruction="""You are a helpful customer support agent for an e-commerce platform.
Use the conversation history below to personalise your response.
History: {history}""",
    before_agent_callback=before_agent,
    after_agent_callback=after_agent,
)

init_db()
session_service = InMemorySessionService()
runner = Runner(agent=agent, app_name="support_app", session_service=session_service)
How ADK callbacks work: before_agent_callback fires before the LLM is called each turn. after_agent_callback fires after the response is returned. This gives you clean hooks to load and save history without touching the agent's core logic. Read more in the Google ADK Part 6: Extras blog.

LangChain + SQLChatMessageHistory

LangChain ships SQLChatMessageHistory which handles table creation, message insertion, and retrieval automatically. You just give it a session ID and a database connection string. 

from langchain_community.chat_message_histories import SQLChatMessageHistory
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_ollama import ChatOllama

# SQLChatMessageHistory creates the table on first use
def get_history(session_id: str):
    return SQLChatMessageHistory(
        session_id=session_id,
        connection_string="sqlite:///support.db"
    )

llm = ChatOllama(model="llama3.2")

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful customer support agent for an e-commerce platform."),
    MessagesPlaceholder(variable_name="history"),  # injected automatically
    ("human", "{input}"),
])

chain_with_history = RunnableWithMessageHistory(
    prompt | llm,
    get_history,
    input_messages_key="input",
    history_messages_key="history",
)

# Each call automatically loads history, appends the new turn, and saves it
response = chain_with_history.invoke(
    {"input": "My order #1234 hasn't arrived yet."},
    config={"configurable": {"session_id": "user_42_session_1"}},
)
print(response.content)

# Second turn - history is loaded automatically from SQLite
response2 = chain_with_history.invoke(
    {"input": "Can you check the status again?"},
    config={"configurable": {"session_id": "user_42_session_1"}},
)
print(response2.content)

6. The Problem with Sync

Both snippets above are synchronous. Every call to sqlite3.connect() and every cursor.fetchall() is a blocking operation: the Python thread sits and waits for the disk to respond before doing anything else.

For a single user this is fine: disk reads are fast and the latency is invisible. But the moment you have multiple users hitting the bot concurrently, the problem becomes clear. Consider this timeline: 

# With sync I/O, users queue behind each other:
# User A  → load_history()  [thread BLOCKED waiting for disk, ~10ms]
# User B  → waiting...
# User C  → waiting...

# With async I/O, the event loop handles all three concurrently:
# User A  → await load_history_async()  [event loop yields, handles B and C]
# User B  → await load_history_async()  [runs concurrently with A]
# User C  → await load_history_async()  [runs concurrently with A and B]

A single sync worker serving 50 concurrent users will have 49 of them waiting at any given moment. Each wait is short individually, but they stack up into noticeable latency. Async eliminates this stacking.

7. Async Upgrade with aiosqlite

aiosqlite is a drop-in async wrapper around the standard sqlite3 library. The API is almost identical; you just add async / await and use it inside an async context manager. 

pip install aiosqlite

Here is the before/after diff for the two key helpers: 

import aiosqlite

# BEFORE (sync)
def load_history(session_id: str) -> list[dict]:
    conn = sqlite3.connect(DB_PATH)
    rows = conn.execute(
        "SELECT role, content FROM messages WHERE session_id=? ORDER BY ts",
        (session_id,)
    ).fetchall()
    conn.close()
    return [{"role": r, "content": c} for r, c in rows]

# AFTER (async) - only 3 words changed: async, await, aiosqlite
async def load_history_async(session_id: str) -> list[dict]:
    async with aiosqlite.connect(DB_PATH) as db:
        async with db.execute(
            "SELECT role, content FROM messages WHERE session_id=? ORDER BY ts",
            (session_id,)
        ) as cursor:
            rows = await cursor.fetchall()
    return [{"role": r, "content": c} for r, c in rows]


# BEFORE (sync)
def save_message(session_id: str, role: str, content: str):
    conn = sqlite3.connect(DB_PATH)
    conn.execute(
        "INSERT INTO messages (session_id, role, content) VALUES (?,?,?)",
        (session_id, role, content)
    )
    conn.commit()
    conn.close()

# AFTER (async)
async def save_message_async(session_id: str, role: str, content: str):
    async with aiosqlite.connect(DB_PATH) as db:
        await db.execute(
            "INSERT INTO messages (session_id, role, content) VALUES (?,?,?)",
            (session_id, role, content)
        )
        await db.commit()

With these two functions async, the ADK callbacks become async def functions and the LangChain chain can be run inside an async context. The event loop can now serve many users concurrently from a single Python process.

8. What We Gained and What Is Still Missing

What the bot can now do

  • Remember everything said in the current conversation.
  • Resume a session even if the server restarts (history is on disk).
  • Handle many users concurrently without blocking (async path).
  • Store history in a structured, queryable format.

What it still cannot do

  • Fast active-session reads. SQLite reads from disk on every turn. For high-frequency chat traffic (>20 messages/min per user) this adds up. Redis (Part 2) keeps the active conversation in RAM for sub-millisecond reads.
  • Cold-resume after long inactivity. If a user comes back three days later, the bot loads the full raw message history from SQLite. This is slow and wastes tokens. Part 2 introduces a rolling window and a running summary so the bot always loads only what it needs.
  • Knowing who you are. The bot has no concept of the user beyond this session. It cannot say “last time you contacted us about a billing issue.” That requires cross-session semantic memory, which we build in Part 3.
SQLite in production: SQLite uses file-level locking. With many concurrent writers you will start seeing database is locked errors. For anything beyond a handful of concurrent users, consider PostgreSQL for the long-term store in Part 2, or move to the Redis-first architecture entirely.

Summary

  • LLMs are stateless by design. Memory is your responsibility.
  • SQLite gives you durable, queryable conversation storage with zero server setup.
  • Two tables (sessions and messages) are all you need for basic history.
  • ADK callbacks (before_agent_callback, after_agent_callback) are the clean way to hook load/save logic around the agent lifecycle.
  • LangChain's SQLChatMessageHistory + RunnableWithMessageHistory automates the entire load/save cycle.
  • Replace sqlite3 with aiosqlite and add async / await to serve many users concurrently without blocking.
  • SQLite works for development and low-traffic apps. For fast active-session reads and cross-session context, continue to Part 2: Short-Term & Long-Term Memory with Redis.
