What Are Workflow Agents?

In the previous post we built a multi-agent system using a plain Agent as the orchestrator — Gemini decided at runtime which sub-agent to delegate to based on the user's message. That approach is flexible and conversational, but sometimes you need deterministic, structured control flow.

That's where Workflow Agents come in. ADK ships three of them:

  • SequentialAgent — runs sub-agents one after the other in a fixed order
  • LoopAgent — repeats sub-agents in a loop until a condition is met
  • ParallelAgent — runs sub-agents concurrently, in parallel

Workflow agents do not use Gemini to decide what to do next — the execution order is hardcoded into the agent definition itself. This makes them predictable, fast, and easy to reason about.

What Is a SequentialAgent?

A SequentialAgent takes a list of sub-agents and runs them one by one, in the exact order you declare them. Each agent's output becomes part of the shared conversation context that the next agent can read from. It is the ADK equivalent of a data pipeline — Step A feeds Step B which feeds Step C.

When to use SequentialAgent:
  • You have a fixed, multi-step process (e.g., research → draft → review)
  • Each step depends on the output of the previous step
  • You need guaranteed execution order, not LLM-decided routing
  • You are building ETL-style pipelines with AI agents

How Context Flows Between Steps

ADK's session maintains a single shared conversation history. When SequentialAgent runs Step 1, the agent's response is appended to this history. Step 2 then receives the full history including Step 1's output as context. Step 3 receives everything Step 1 and Step 2 produced, and so on.

For more structured data sharing, sub-agents can use tools that read from and write to tool_context.state — a dictionary that persists across all agents in a session. This is the recommended pattern when agents need to pass structured results (JSON, lists, numbers) rather than just natural language. 


# State is a dict on tool_context that all agents in a session share
def save_to_state(key: str, value: str, tool_context: ToolContext) -> dict:
    tool_context.state[key] = value
    return {"status": "saved"}

def read_from_state(key: str, tool_context: ToolContext) -> dict:
    return {"value": tool_context.state.get(key, "")}

Step 2: Define the Tools

Each agent in our pipeline needs at least one tool to perform its job. We also need shared-state helpers so agents can hand off structured data to each other. 


# agent.py
import os
from dotenv import load_dotenv
from google.adk.agents import Agent, SequentialAgent
from google.adk.tools import ToolContext

load_dotenv()

# ── Shared state helpers ──────────────────────────────────────────────

def save_research(facts: str, tool_context: ToolContext) -> dict:
    """Save research facts to shared session state for downstream agents.

    Args:
        facts: A string containing the research findings.
        tool_context: ADK tool context providing access to session state.

    Returns:
        A dict confirming the save operation.
    """
    tool_context.state["research_facts"] = facts
    return {"status": "research saved", "char_count": len(facts)}


def load_research(tool_context: ToolContext) -> dict:
    """Load previously saved research facts from session state.

    Args:
        tool_context: ADK tool context providing access to session state.

    Returns:
        A dict containing the research facts.
    """
    facts = tool_context.state.get("research_facts", "No research found.")
    return {"facts": facts}


def save_draft(draft: str, tool_context: ToolContext) -> dict:
    """Save the written draft to shared session state.

    Args:
        draft: The written blog draft.
        tool_context: ADK tool context providing access to session state.

    Returns:
        A dict confirming the save.
    """
    tool_context.state["draft"] = draft
    return {"status": "draft saved", "char_count": len(draft)}


def load_draft(tool_context: ToolContext) -> dict:
    """Load the previously saved draft from session state.

    Args:
        tool_context: ADK tool context providing access to session state.

    Returns:
        A dict containing the draft text.
    """
    draft = tool_context.state.get("draft", "No draft found.")
    return {"draft": draft}

Step 3: Build the Three Sub-Agents

Each sub-agent has a focused, single-purpose instruction. Notice how the writer and editor agents are told to use the shared state tools — this is the key to passing structured data through the pipeline. 


# ── Sub-agent 1: Researcher ───────────────────────────────────────────

researcher_agent = Agent(
    name="researcher_agent",
    model="gemini-2.0-flash",
    description="Researches a topic and saves facts to shared state.",
    instruction=(
        "You are a research assistant. When given a topic, call the fetch_facts tool "
        "to gather information about it. Then call save_research to store those facts "
        "in shared state so the next agent can use them. "
        "Do not write the blog post — just collect and save the facts."
    ),
    tools=[fetch_facts, save_research],
)


# ── Sub-agent 2: Writer ───────────────────────────────────────────────

writer_agent = Agent(
    name="writer_agent",
    model="gemini-2.0-flash",
    description="Writes a blog draft based on research facts from shared state.",
    instruction=(
        "You are a technical blog writer. Call load_research to retrieve the facts "
        "collected by the researcher. Use those facts to write a clear, engaging "
        "200-300 word blog section. Then call save_draft to store your draft in "
        "shared state. Do not edit the draft yourself — just write and save it."
    ),
    tools=[load_research, save_draft],
)


# ── Sub-agent 3: Editor ───────────────────────────────────────────────

editor_agent = Agent(
    name="editor_agent",
    model="gemini-2.0-flash",
    description="Edits and polishes the draft retrieved from shared state.",
    instruction=(
        "You are a senior technical editor. Call load_draft to retrieve the draft "
        "written by the writer agent. Review it for clarity, grammar, and tone. "
        "Fix any awkward phrasing. Make it concise and professional. "
        "Output the final polished version directly as your response."
    ),
    tools=[load_draft],
)

Step 4: Wire Them Into a SequentialAgent

Now we wrap all three agents in a SequentialAgent. The sub_agents list defines the exact execution order — ADK will run them left to right, no LLM routing involved. 


# ── SequentialAgent pipeline ──────────────────────────────────────────

content_pipeline = SequentialAgent(
    name="content_pipeline",
    description=(
        "A 3-step content creation pipeline: research a topic, write a draft, "
        "then edit and polish it. Runs researcher → writer → editor in order."
    ),
    sub_agents=[researcher_agent, writer_agent, editor_agent],
)

root_agent = content_pipeline

Key Concepts to Remember

Concept Detail
Execution order Always left-to-right in the sub_agents list. Not LLM-decided.
Context sharing Each agent sees the full conversation history, including all prior agents' outputs.
Structured data sharing Use tool_context.state (a dict) to pass structured data between steps.
Agent instructions Each agent's instruction should be scoped to its single responsibility.
SequentialAgent has no LLM The pipeline itself does not use Gemini. Only the sub-agents do.
root_agent The SequentialAgent itself is the entry point — set it as root_agent.

SequentialAgent vs Orchestrator Agent

A common question is when to use a SequentialAgent versus a plain Agent with sub_agents (the pattern from Part 2).

Orchestrator Agent (Part 2) SequentialAgent (Part 3)
Routing Gemini decides dynamically based on user input Hardcoded order, always the same
Flexibility High — adapts to different inputs Low — fixed pipeline
Predictability Lower — depends on LLM judgment High — deterministic
Best for Conversational assistants, task delegation ETL pipelines, document processing, multi-step workflows

Final Thoughts

The SequentialAgent gives you assembly-line power for your AI workflows. Once you need guaranteed step ordering — whether it's research → summarise → translate, or validate → enrich → store — reach for SequentialAgent instead of hoping your LLM orchestrator picks the right path.

The shared state pattern via tool_context.state is particularly valuable: it lets you move rich, structured data between agents without embedding it all in natural language — keeping your outputs clean and your pipeline robust.
