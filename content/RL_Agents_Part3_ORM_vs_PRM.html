1. The RL Optimization Problem

Once you have reward signals (from RLVR), the question becomes: how do you adjust the model's weights to produce higher-reward trajectories?

The classical approach is PPO (Proximal Policy Optimization), which works like this:

  1. Sample trajectories from your current model
  2. Compute advantages (how much better is this trajectory than average?)
  3. Use gradient updates to increase the probability of high-advantage trajectories
  4. Regularize: use KL divergence to keep the updated model close to the original (don't drift too far)

PPO works well, but it has a significant drawback: computing advantages requires a separate critic network - another neural network that predicts expected returns. For large models, this means duplicating the model size, doubling computation and memory.

2. GRPO: Group Relative Policy Optimization

GRPO (Group Relative Policy Optimization) is a clever alternative that sidesteps the critic network entirely. The core insight: instead of comparing each trajectory to a learned baseline, compare outputs within a group.

How GRPO Works:
  1. Sample a batch of outputs: For a single prompt, generate multiple responses (say, 8 candidate solutions).
  2. Score them all: Use your reward model (or verifier) to score each response.
  3. Compute relative advantages: Instead of comparing to an expected value, compute advantages relative to the group mean. High scorer gets positive advantage. Low scorer gets negative advantage.
  4. Update the policy: Increase probability of high-advantage responses, decrease low-advantage responses.

Why is this efficient?

  • No critic network: You're using the group mean as your baseline instead of a learned function. Saves 2x compute.
  • Natural distribution: By sampling multiple outputs per prompt, you efficiently explore the space around your current policy.
  • Empirically strong: DeepSeek-R1 used GRPO and achieved state-of-the-art results with significantly lower computational cost than expected.
Simple Example:

Suppose you're training a math-solving agent. For the prompt "What is 5 + 3?", you generate 4 candidate answers:

  • Candidate A: "The answer is 8" → Verifier checks: correct! Reward = 1
  • Candidate B: "The answer is 9" → Verifier checks: incorrect. Reward = 0
  • Candidate C: "The answer is 8" → Verifier checks: correct! Reward = 1
  • Candidate D: "The answer is 10" → Verifier checks: incorrect. Reward = 0

Group mean reward = (1 + 0 + 1 + 0) / 4 = 0.5

Advantages: A gets +0.5, B gets -0.5, C gets +0.5, D gets -0.5

During the gradient update, you increase the probability of generating responses like A and C, and decrease the probability of responses like B and D. This is much cheaper than training a critic network.

3. Monte Carlo Tree Search + LLMs

An orthogonal approach: instead of pure RL fine-tuning, use MCTS (Monte Carlo Tree Search) at inference time to explore the space of reasoning paths.

Classic MCTS in Games:

MCTS is the algorithm that powers AlphaGo. It works by:

  1. Selection: Starting from the root (current state), pick the most promising path using an exploration strategy (UCB1 formula).
  2. Expansion: When you reach a leaf node, add new children (new possible moves).
  3. Simulation: Run a fast simulation from the expanded node to estimate its value.
  4. Backpropagation: Update all nodes along the path with the simulation result.
Applying MCTS to Reasoning:

For agent tasks, the analogy is:

  • Moves: The next action or reasoning step the agent could take.
  • Game state: The current context, partial solution, or partially-completed task.
  • Value function: A neural network (or another LLM) that scores whether a state looks like it's on the path to a solution.

The agent generates candidate next steps (using the language model), MCTS explores them using the value function, and the path with the highest estimated value is executed.

Example: Solving a Math Problem with MCTS

Prompt: "Solve for x: 3x + 5 = 20"

  1. Root: Initial state is the problem statement.
  2. MCTS samples steps:
    • Path A: "Subtract 5 from both sides" → "3x = 15"
    • Path B: "Divide both sides by 3" → "x + 5/3 = 20/3" (wrong approach)
    • Path C: "Subtract 5 from both sides" → "3x = 15" (same as A)
  3. Value function evaluates each: Paths A and C look promising (score 0.9). Path B looks like a dead end (score 0.3).
  4. MCTS explores further: Expands path A/C: "Divide both sides by 3" → "x = 5"
  5. Final step: Verify "x = 5" against the original equation. It works! Done.
Why MCTS for Reasoning?
  • Explicit search: You explore multiple paths before committing, not just sampling once from the LLM.
  • Efficient exploration: UCB1 balances exploring promising paths vs exploring new alternatives.
  • Generalizes to complex problems: Beam search (a simpler version) and full MCTS handle much longer reasoning chains.
  • State-of-the-art results: AlphaProof (DeepMind's math olympiad solver) used MCTS-like search and achieved silver-medal performance.

4. GRPO vs MCTS: When to Use Which?

Aspect GRPO (RL Fine-tuning) MCTS (Inference Search)
When applied During training At inference (test) time
Cost Amortized during training Per-query (can be slow)
How it helps Model learns better policies Explores best solutions at test time
Requires Verifiable reward signal Verifiable reward + value function
Best for Scalable improvement, batch processing Complex problems needing explicit search

The future approach: Use both. Fine-tune with GRPO to improve the base model. Then add MCTS at inference time for extra-hard problems where search helps.

5. The Branching Factor Problem in Language

MCTS is incredibly powerful, but it faces a challenge in language: the branching factor is huge. In Go, each position has ~300 legal moves. In language generation, each token could be any of ~100k vocabulary items, leading to combinatorial explosion.

Solutions:

  • Beam search: Restrict to top-K candidates at each step (K=3 or K=5). Much cheaper than full MCTS.
  • Constraint-guided generation: Only allow moves (tokens) that maintain validity (e.g., don't break syntax).
  • Value-guided pruning: Discard branches with low value predictions early.

This is why AlphaProof worked: it combined a strong language model (proposing reasonable steps) with a value function (filtering implausible paths) and beam search (exploring top candidates). Not full MCTS, but the principle is the same.

6. Real-World Impact

The models leading the leaderboards now use some blend of:

  • GRPO (or similar) during training: Efficient fine-tuning without doubling model size.
  • Search at inference: Beam search or MCTS for hard problems, direct generation for easy ones (amortized cost).
  • Value functions: Trained to predict solution likelihood, guiding search.

DeepSeek-R1's success came largely from this exact stack: GRPO training + test-time inference search. The cost tradeoff is real (inference is slower), but the capability jump is enormous.

7. Complete Code: GRPO and MCTS Demonstrations

The code is organized into three simulation blocks so you can map each algorithm to its real runtime behavior.

  • GRPO block: Builds a candidate-response group, scores each response, and computes relative advantages to show which outputs get reinforced.
  • MCTS block: Simulates node selection with UCB, showing how search balances high-value paths with under-explored paths.
  • Beam search block: Expands top candidate traces over multiple steps, imitating inference-time reasoning path exploration.
import math
   import random
   import pandas as pd

   # PART 1: GRPO - Group Relative Policy Optimization

   def ppo_clipped_objective(old_prob: float, new_prob: float, advantage: float, eps: float = 0.2) -> float:
     """
     Classical PPO objective (for reference).
     Clips ratio to prevent too-large updates.
     """
     ratio = new_prob / max(old_prob, 1e-9)
     clipped_ratio = min(max(ratio, 1 - eps), 1 + eps)
     # Return the minimum of original and clipped, times advantage
     return min(ratio * advantage, clipped_ratio * advantage)

   def grpo_relative_advantages(rewards: list[float]) -> list[float]:
     """
     GRPO: Compute advantages relative to group mean (not a learned baseline).
     """
     group_mean = sum(rewards) / max(len(rewards), 1)
     advantages = [round(r - group_mean, 3) for r in rewards]
     return advantages

   # Demo: GRPO on a simple problem
   print("=" * 70)
   print("GRPO: Group Relative Policy Optimization")
   print("=" * 70)

   # Scenario: Generate 6 solutions to "What is 2 + 3?"
   rewards = [1.0, 0.0, 1.0, 0.0, 0.5, 0.8]  # 6 candidate outputs scored by verifier
   candidate_names = ["Sol_A", "Sol_B", "Sol_C", "Sol_D", "Sol_E", "Sol_F"]

   print("\nCandidate solutions and their verifier scores:")
   for name, reward in zip(candidate_names, rewards):
     print(f"  {name}: {reward}")

   group_mean = sum(rewards) / len(rewards)
   print(f"\nGroup mean reward: {group_mean:.2f}")

   advantages = grpo_relative_advantages(rewards)
   print("\nGRPO Advantages (relative to group mean):")
   for name, adv in zip(candidate_names, advantages):
     direction = "↑ increase prob" if adv > 0 else "↓ decrease prob"
     print(f"  {name}: {adv:+.3f} {direction}")

   # PART 2: MCTS - UCB-Based Selection

   def ucb_score(win_rate: float, visits: int, total_visits: int, c: float = 1.4) -> float:
     """
     UCB1 formula: balances exploitation (win_rate) and exploration (math.log term).
     """
     if visits == 0:
       return float("inf")  # Unvisited nodes get priority

     exploitation = win_rate
     exploration = c * math.sqrt(math.log(total_visits) / visits)
     return exploitation + exploration

   def mcts_lite_selection(seed: int = 5) -> pd.DataFrame:
     """
     Simulate MCTS selection: which path should we explore next?
     Imagine 4 candidate next steps, each with different visit counts and values.
     """
     rng = random.Random(seed)

     paths = []
     total_visits = 50

     for path_name in ["path_a", "path_b", "path_c", "path_d"]:
       visits = rng.randint(1, 20)
       win_rate = rng.uniform(0.1, 0.95)
       ucb = ucb_score(win_rate, visits, total_visits)

       paths.append({
         "path": path_name,
         "win_rate": round(win_rate, 3),
         "visits": visits,
         "ucb_score": round(ucb, 3)
       })

     frame = pd.DataFrame(paths)
     return frame.sort_values("ucb_score", ascending=False)

   # Demo: MCTS Selection
   print("\n" + "=" * 70)
   print("MCTS: Monte Carlo Tree Search - Path Selection")
   print("=" * 70)

   print("\nAfter 50 total simulations, 4 candidate next steps have different statistics:")
   mcts_frame = mcts_lite_selection()
   print(mcts_frame.to_string(index=False))

   print("\nInterpretation:")
   print("- path_a: high win rate but fewer visits")
   print("- path_b: moderate stats, moderate UCB")
   print("- path_c: good win rate with decent visits → high UCB (explore this!)")
   print("- path_d: low win rate → low UCB (avoid)")

   print("\nUCB selection priority:")
   for idx, (path, ucb) in enumerate(zip(mcts_frame["path"], mcts_frame["ucb_score"]), 1):
     print(f"  {idx}. {path} (UCB={ucb})")

   # PART 3: Beam Search Example

   def beam_search_arithmetic(target: int = 15, beam_width: int = 3, depth: int = 4) -> pd.DataFrame:
     """
     Simplified beam search: find a sequence of arithmetic operations to reach target.
     Example: start=0, +3, +3, +3, +3, +3 = 15
     """
     # Start with initial state
     beams = [(0, "start")]
     operations = [("+1", 1), ("+3", 3), ("*2", 2)]  # Available operations

     # Expand for `depth` iterations
     for iteration in range(depth):
       all_candidates = []

       for value, trace in beams:
         for op_name, op_value in operations:
           # Apply operation
           if op_name == "*2":
             next_value = value * op_value
           else:
             next_value = value + op_value

           new_trace = f"{trace} -> {op_name}({next_value})"
           all_candidates.append((next_value, new_trace))

       # Keep only top-K by distance to target
       all_candidates.sort(key=lambda x: abs(target - x[0]))
       beams = all_candidates[:beam_width]

     # Return final results
     results = []
     for value, trace in beams:
       results.append({
         "value": value,
         "distance_to_target": abs(target - value),
         "trace": trace
       })

     return pd.DataFrame(results).sort_values("distance_to_target")

   print("\n" + "=" * 70)
   print("BEAM SEARCH: Arithmetic Path Finding")
   print("=" * 70)

   print("\nFind operations to reach target=15 (starting from 0)")
   beam_results = beam_search_arithmetic(target=15, beam_width=3, depth=4)
   print("\nTop beams after 4 steps:")
   for idx, row in beam_results.iterrows():
     print(f"\nValue: {row['value']}, Distance: {row['distance_to_target']}")
     print(f"Trace: {row['trace']}")

   print("\n" + "=" * 70)
   print("KEY INSIGHTS:")
   print("=" * 70)
   print("1. GRPO avoids critic networks by using group-relative advantages")
   print("2. MCTS uses UCB to balance exploitation and exploration")
   print("3. Beam search is a practical variant: keep top-K, expand, repeat")
   print("4. Combined: GRPO trains the policy, MCTS explores at inference")
   print("=" * 70)

When you run this code, you'll see:

  • GRPO advantages show which solutions should be reinforced (positive) vs discouraged (negative)
  • MCTS UCB scores prioritize well-performing paths while still exploring newcomers
  • Beam search efficiently finds near-optimal arithmetic sequences by exploring promising branches
