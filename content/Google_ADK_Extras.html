1. Connecting Ollama and Other LLMs via LiteLLM

Why Use a Different LLM?

ADK uses Gemini by default, but sometimes you need a different model for cost, privacy, latency, or capability reasons. You might want to run Llama 3 locally with Ollama during development to avoid API costs, use GPT-4o for a task where it excels, or plug in a fine-tuned model hosted on your own infrastructure. ADK supports all of this through its LiteLLM integration.

LiteLLM is an open-source Python library that provides a unified interface to over 100 LLM providers. ADK wraps it as a first-class model option via the LiteLlm class.

Installation


# Install ADK with LiteLLM support
pip install google-adk[litellm]

# If you want to use Ollama locally, install it separately:
# https://ollama.com/download
# Then pull a model:
ollama pull llama3.2

Using Ollama (Local Models)

Ollama runs a local server at http://localhost:11434. LiteLLM can talk to it using the ollama/ model prefix. This means your agent never makes an outbound API call everything stays on your machine. 


from google.adk.agents import Agent
from google.adk.models.lite_llm import LiteLlm

# Make sure Ollama is running: `ollama serve`
# And the model is pulled: `ollama pull llama3.2`

agent = Agent(
    name="local_agent",
    model=LiteLlm(model="ollama/llama3.2"),
    description="A locally-running agent powered by Llama 3.2 via Ollama.",
    instruction="You are a helpful assistant. Answer concisely.",
    tools=[],
)

root_agent = agent

2. Session Memory, State, and Traces

Understanding ADK's Memory Layers

ADK has three distinct levels of memory, each with a different scope and lifetime. Understanding which one to use is critical for building agents that behave correctly across conversations.

Layer Scope Lifetime How to access
In-turn state Current tool call only Wiped after the tool returns tool_context.state (temporary writes)
Session state One conversation session Until the session ends tool_context.state (persists across turns within a session)
Long-term memory Across multiple sessions Permanent (until deleted) MemoryService explicitly save/query

Full runnable code: 04_agent_with_session_state.py

Session Services

A SessionService manages how sessions are stored. ADK provides three built-in implementations.

InMemorySessionService (default for adk run / adk web)

Stores everything in process memory. Fast, no setup required. Sessions are lost when the process restarts. Good for development and testing. 


from google.adk.sessions import InMemorySessionService
from google.adk.runners import Runner

session_service = InMemorySessionService()

runner = Runner(
    agent=root_agent,
    app_name="my_app",
    session_service=session_service,
)
DatabaseSessionService (persistent local storage)

Persists sessions to a SQLite database (or any SQLAlchemy-compatible database). Sessions survive process restarts. Use this for local development when you need persistence. 


from google.adk.sessions import DatabaseSessionService
from google.adk.runners import Runner

# SQLite for local development
session_service = DatabaseSessionService(db_url="sqlite:///./sessions.db")

# PostgreSQL for production
# session_service = DatabaseSessionService(
#     db_url="postgresql://user:password@localhost:5432/mydb"
# )

runner = Runner(
    agent=root_agent,
    app_name="my_app",
    session_service=session_service,
)
VertexAiSessionService (managed cloud storage)

Stores sessions in Vertex AI's managed infrastructure. The right choice when deploying to GCP no database to manage, sessions are automatically persisted, scaled, and secured. 


from google.adk.sessions import VertexAiSessionService
from google.adk.runners import Runner

session_service = VertexAiSessionService(
    project="my-gcp-project",
    location="us-central1",
)

runner = Runner(
    agent=root_agent,
    app_name="my_app",
    session_service=session_service,
)

Using the Runner Programmatically

When you use adk run or adk web, ADK manages the Runner for you. For production deployments or when you want full control you construct it yourself. 


from google.adk.sessions import InMemorySessionService
from google.adk.runners import Runner
from google.genai import types as genai_types

session_service = InMemorySessionService()

# Create a session for a user
session = session_service.create_session(
    app_name="my_app",
    user_id="user_123",
)

runner = Runner(
    agent=root_agent,
    app_name="my_app",
    session_service=session_service,
)

# Build the user message
message = genai_types.Content(
    role="user",
    parts=[genai_types.Part(text="What is the weather in Paris?")]
)

# Run the agent  returns an event stream
for event in runner.run(
    user_id="user_123",
    session_id=session.id,
    new_message=message,
):
    if event.is_final_response():
        print("Agent:", event.content.parts[0].text)

Long-Term Memory with MemoryService

Session state only lasts for one conversation. For information that should persist across multiple sessions user preferences, past interactions, accumulated knowledge use the MemoryService. 


from google.adk.memory import InMemoryMemoryService
from google.adk.runners import Runner

memory_service = InMemoryMemoryService()

runner = Runner(
    agent=root_agent,
    app_name="my_app",
    session_service=InMemorySessionService(),
    memory_service=memory_service,    # attach memory service here
)

Tools can then read from and write to long-term memory using tool_context.search_memory() and by adding memories via the session's memory_service. 


from google.adk.tools import ToolContext

def remember_preference(preference: str, tool_context: ToolContext) -> dict:
    """Save a user preference to long-term memory.

    Args:
        preference: The preference to remember (e.g., 'prefers bullet points').
        tool_context: ADK tool context.

    Returns:
        Confirmation dict.
    """
    # tool_context.state persists within this session only
    tool_context.state["last_preference"] = preference
    return {"status": "noted", "preference": preference}


def recall_context(query: str, tool_context: ToolContext) -> dict:
    """Search long-term memory for relevant past context.

    Args:
        query: What to search for in memory.
        tool_context: ADK tool context.

    Returns:
        Dict with matching memory entries.
    """
    results = tool_context.search_memory(query)
    return {"memories": [r.content for r in results]}

Traces and Observability

ADK emits structured events for every agent call, tool call, and model invocation. These events form a trace that lets you inspect exactly what happened during a run.

Reading Events from the Runner

for event in runner.run(user_id="u1", session_id=session.id, new_message=message):
    # Every event has a type you can inspect
    print(f"Event type: {event.__class__.__name__}")

    if event.content:
        for part in event.content.parts:
            if hasattr(part, "text") and part.text:
                print(f"  Text: {part.text[:80]}")
            if hasattr(part, "function_call") and part.function_call:
                print(f"  Tool call: {part.function_call.name}({part.function_call.args})")
            if hasattr(part, "function_response") and part.function_response:
                print(f"  Tool result: {part.function_response.response}")

    if event.is_final_response():
        print("=== Final answer ===")
        print(event.content.parts[0].text)
Google Cloud Trace Integration

When deployed on GCP, ADK automatically sends traces to Cloud Trace. No extra configuration needed every agent invocation becomes a traceable span. You can view latency breakdowns, tool call timings, and model response times directly in the GCP console under Trace → Trace List.

For local development, you can also export traces to any OpenTelemetry-compatible backend (Jaeger, Zipkin, etc.) by configuring the standard OpenTelemetry SDK before initialising ADK. 


# Enable OpenTelemetry tracing locally (optional)
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import ConsoleSpanExporter, SimpleSpanProcessor

provider = TracerProvider()
provider.add_span_processor(SimpleSpanProcessor(ConsoleSpanExporter()))
trace.set_tracer_provider(provider)

# Now run your agent  trace spans will print to console

3. Deploying to Google Cloud Platform

Two Deployment Options

ADK provides two adk deploy targets:

  • Cloud Run deploy your agent as a containerised REST service. You own the container, you control scaling. Good for custom infrastructure, complex dependencies, or cost control.
  • Vertex AI Agent Engine fully managed ADK runtime hosted by Google. No containers, no infrastructure management. Google handles scaling, session storage, and availability. The fastest path to production.

Prerequisites

Option A: Deploy to Cloud Run

Cloud Run wraps your agent in an HTTP container. ADK handles the Dockerfile and container build automatically. Your agent becomes a REST endpoint that any client can call.

  4. Print the public HTTPS URL of your agent service

Option B: Deploy to Vertex AI Agent Engine

Agent Engine is Google's fully managed runtime for ADK agents. It handles session persistence, scaling, monitoring, and authentication automatically. You upload your agent code; Google runs it.

4. Analysing Model Performance with ADK Eval

Why Evaluate?

An agent that works in manual testing does not always work reliably on real traffic. ADK's built-in evaluation framework lets you write a set of test cases with expected tool calls and expected final answers and run them automatically. This tells you:

  • Is my agent calling the right tools with the right arguments?
  • Is the final answer semantically close to the expected answer?
  • Which queries does my agent reliably handle and which does it fail?

Writing an Eval Dataset

Eval data is a JSON file. Each entry has a query (the user input), an optional expected_tool_use list (what tools should be called), and an optional reference (the ideal final answer for semantic comparison). 


// eval_data.json
[
  {
    "query": "What is 15 multiplied by 7?",
    "expected_tool_use": [
      {
        "tool_name": "calculate",
        "tool_input": {"expression": "15 * 7"}
      }
    ],
    "reference": "105"
  },
  {
    "query": "What is the weather in London?",
    "expected_tool_use": [
      {
        "tool_name": "get_weather",
        "tool_input": {"city": "London"}
      }
    ],
    "reference": "The current weather in London is partly cloudy with a temperature of 14°C."
  },
  {
    "query": "Tell me a joke.",
    "expected_tool_use": [],
    "reference": "A short, funny joke."
  }
]

Running the Evaluation


# Basic eval run
adk eval my_agent_folder/ eval_data.json

# Save results to a file
adk eval my_agent_folder/ eval_data.json --output=eval_results.json

# Run with verbose output (shows each query result)
adk eval my_agent_folder/ eval_data.json --verbose

Eval Metrics Explained

Metric What it measures Range
Tool use accuracy Did the agent call the right tool with the right arguments? 0–100% (exact match)
Response similarity How close is the agent's answer to the reference? Uses ROUGE-L by default. 0.0–1.0
Trajectory match Did the agent take the expected sequence of tool calls, in order? 0–100%

Programmatic Evaluation

For CI/CD pipelines you can run evals in Python, check scores programmatically, and fail a deployment if quality drops below a threshold. 


from google.adk.evaluation import AgentEvaluator

evaluator = AgentEvaluator()

# Load your test cases
test_cases = [
    {
        "query": "What is 15 * 7?",
        "expected_tool_use": [{"tool_name": "calculate", "tool_input": {"expression": "15 * 7"}}],
        "reference": "105",
    },
]

# Run evaluation
results = evaluator.evaluate(
    agent_module="my_agent_folder",
    eval_dataset=test_cases,
)

# Check results and fail CI if below threshold
tool_accuracy = results["tool_use_accuracy"]
response_sim = results["response_similarity"]

print(f"Tool accuracy: {tool_accuracy:.1%}")
print(f"Response similarity: {response_sim:.2f}")

if tool_accuracy < 0.90 or response_sim < 0.75:
    raise SystemExit("Agent quality below threshold  deployment blocked.")

5. Other ADK Features Worth Knowing

Callbacks Intercept Every Step

Callbacks are hook functions that fire before or after each major ADK event: before the model is called, after the model responds, before a tool runs, or after it returns. Use them for logging, validation, rate limiting, cost tracking, or modifying inputs and outputs on the fly.

before_model_callback inspect or modify the LLM request

from google.adk.agents.callback_context import CallbackContext
from google.adk.models import LlmRequest, LlmResponse
import time

def log_model_request(
    callback_context: CallbackContext,
    llm_request: LlmRequest,
) -> LlmResponse | None:
    """Log every model call with timing. Return None to let the call proceed."""
    num_messages = len(llm_request.contents) if llm_request.contents else 0
    print(f"[{time.strftime('%H:%M:%S')}] Calling model  {num_messages} messages in context")
    return None   # returning None means: proceed normally


agent = Agent(
    name="my_agent",
    model="gemini-2.0-flash",
    description="Agent with model call logging.",
    instruction="You are a helpful assistant.",
    tools=[],
    before_model_callback=log_model_request,
)
before_model_callback short-circuit the model call entirely

If your callback returns an LlmResponse instead of None, ADK skips the actual model call and uses your response instead. This is useful for caching, guardrails, or injecting canned responses during testing. 


from google.adk.agents.callback_context import CallbackContext
from google.adk.models import LlmRequest, LlmResponse
from google.genai import types as genai_types

BLOCKED_TOPICS = ["competitor pricing", "confidential"]

def safety_guardrail(
    callback_context: CallbackContext,
    llm_request: LlmRequest,
) -> LlmResponse | None:
    """Block requests that contain sensitive topics."""
    last_message = ""
    if llm_request.contents:
        last = llm_request.contents[-1]
        if last.parts:
            last_message = last.parts[0].text.lower()

    for topic in BLOCKED_TOPICS:
        if topic in last_message:
            # Return a canned refusal  model is never called
            return LlmResponse(
                content=genai_types.Content(
                    role="model",
                    parts=[genai_types.Part(text="I can't help with that topic.")],
                )
            )
    return None   # safe  proceed normally


agent = Agent(
    name="safe_agent",
    model="gemini-2.0-flash",
    description="Agent with safety guardrails.",
    instruction="You are a helpful assistant.",
    tools=[],
    before_model_callback=safety_guardrail,
)
before_tool_callback validate or transform tool arguments

from google.adk.tools.base_tool import BaseTool
from google.adk.tools.tool_context import ToolContext

def validate_tool_args(
    tool: BaseTool,
    args: dict,
    tool_context: ToolContext,
) -> dict | None:
    """Validate tool arguments before execution. Return None to proceed."""
    # Block dangerous expressions in the calculator tool
    if tool.name == "calculate":
        expr = args.get("expression", "")
        if any(word in expr for word in ["import", "exec", "eval", "open"]):
            # Return a dict to skip the tool call and use this as the result
            return {"error": "Expression contains disallowed keywords."}
    return None   # all good  run the tool normally


agent = Agent(
    name="safe_calc_agent",
    model="gemini-2.0-flash",
    description="Calculator agent with argument validation.",
    instruction="Help with maths.",
    tools=[calculate],
    before_tool_callback=validate_tool_args,
)

Artifacts Storing Files and Binary Data

ADK's artifact system lets agents save and retrieve files generated PDFs, images, CSV exports, audio clips within a session. Artifacts are stored outside the conversation history so they don't bloat the context window. 


from google.adk.tools import ToolContext
import google.genai.types as genai_types

def generate_report(data: str, tool_context: ToolContext) -> dict:
    """Generate a text report and save it as an artifact.

    Args:
        data: Raw data to include in the report.
        tool_context: ADK tool context.

    Returns:
        Dict with artifact filename.
    """
    report_content = f"# Generated Report\n\n{data}\n\nGenerated at: 2025"
    artifact = genai_types.Part.from_text(report_content)

    version = tool_context.save_artifact("report.txt", artifact)
    return {"status": "saved", "filename": "report.txt", "version": version}


def load_report(tool_context: ToolContext) -> dict:
    """Load a previously saved report artifact.

    Args:
        tool_context: ADK tool context.

    Returns:
        Dict with report content.
    """
    artifact = tool_context.load_artifact("report.txt")
    if artifact is None:
        return {"error": "No report found."}
    return {"content": artifact.text}

Artifacts support versioning every save_artifact call increments the version number. You can load a specific version or always get the latest.

Authentication for External Tools

When your tools need to call external APIs that require authentication OAuth 2.0, API keys, service accounts ADK provides a structured way to handle credentials via ToolAuthContext. This keeps credentials out of your tool code and out of the conversation history. 


import os
from google.adk.tools import ToolContext

def call_external_api(endpoint: str, tool_context: ToolContext) -> dict:
    """Call an external API using a stored API key.

    Args:
        endpoint: The API endpoint path to call.
        tool_context: ADK tool context.

    Returns:
        API response dict.
    """
    import requests

    # Read the API key from environment (set via Secret Manager in production)
    api_key = os.environ.get("EXTERNAL_API_KEY", "")
    if not api_key:
        return {"error": "EXTERNAL_API_KEY not configured."}

    headers = {"Authorization": f"Bearer {api_key}"}
    response = requests.get(f"https://api.example.com/{endpoint}", headers=headers)
    return response.json()
Best practice: Load API keys from environment variables (or Secret Manager on GCP). Never hardcode keys in your agent code or return them in tool responses ADK logs tool responses and they can appear in traces.

Structured Output with output_schema

By default, agents return free-form text. If your downstream system needs structured JSON for example, to feed into a database or another API set output_schema on the agent. ADK instructs Gemini to return a response conforming to the schema. 


from pydantic import BaseModel
from google.adk.agents import Agent


class WeatherReport(BaseModel):
    city: str
    temperature_celsius: float
    condition: str
    humidity_percent: int
    recommendation: str


structured_agent = Agent(
    name="structured_weather_agent",
    model="gemini-2.0-flash",
    description="Returns weather as structured JSON.",
    instruction=(
        "When asked about weather, return a structured report with the city name, "
        "temperature, condition, humidity percentage, and a brief recommendation."
    ),
    output_schema=WeatherReport,   # Gemini will format its response as WeatherReport JSON
    tools=[],
)

root_agent = structured_agent

The agent's response will be a WeatherReport instance you can use directly in your code. Note: when output_schema is set, the agent cannot call tools it is a generation-only mode.

Final Thoughts

This post covered the production surface area of ADK that doesn't fit neatly into the agent graph itself. To summarise the key decisions you will make when taking an ADK project from prototype to production:

  • LLM choice: Use Gemini by default. Swap to LiteLlm for Ollama (local/free), OpenAI, Claude, or any other provider. Mix models in multi-agent systems to optimise cost vs capability.
  • Memory: InMemorySessionService for dev, DatabaseSessionService for local persistence, VertexAiSessionService for production on GCP. Use MemoryService only when you need cross-session recall.
  • Deployment: adk deploy cloud_run for full container control; adk deploy agent_engine for the fastest, most managed path. Always use Secret Manager for credentials.
  • Quality: Write an eval dataset early. Run adk eval as part of your CI pipeline. A 90%+ tool accuracy and 0.75+ response similarity is a reasonable baseline.
  • Observability: Use callbacks for logging, cost tracking, and guardrails. Cloud Trace is automatic on GCP. Parse the event stream from Runner.run() for custom metrics.
  • Structured output: Use output_schema when downstream systems need JSON. Use tool_context.save_artifact for generated files.
