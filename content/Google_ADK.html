Google ADK — Building AI Agents with Google's Agent Development Kit

← Back to Home

Google's Agent Development Kit (ADK) is an open-source Python framework for building, evaluating, and deploying AI agents powered by Gemini models. Unlike higher-level wrappers, ADK gives you direct control over how your agent reasons, what tools it can call, and how it manages conversation state — all with minimal boilerplate.

If you've worked with LangChain or LangGraph before, ADK will feel familiar in spirit but is tightly integrated with Google's ecosystem (Gemini, Vertex AI, Google Search). This post covers the essentials to get you up and running:

  • Installing ADK and setting up credentials
  • Project structure and key CLI commands
  • Building a basic agent
  • Writing a custom tool and wiring it to the agent
  • Running your agent via adk run and adk web

1. Installation

ADK requires Python 3.9 or higher. Create a dedicated virtual environment first to keep things clean: 

# Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate      # macOS / Linux
# .venv\Scripts\activate       # Windows

# Install ADK
pip install google-adk

ADK uses Google Gemini models under the hood. You'll need a Gemini API key. Create a .env file at the root of your project: 

# .env
GOOGLE_GENAI_USE_VERTEXAI=FALSE
GOOGLE_API_KEY=your_api_key_here

You can get a free Gemini API key from Google AI Studio. ADK automatically picks up the .env file when you run it from the same directory.

2. Project Structure & ADK Commands

ADK expects a specific folder layout so its CLI knows where to find your agent. The convention is one folder per agent, sitting inside a parent project directory: 

my_project/
├── .env                  # API keys — never commit this
└── weather_agent/        # one folder per agent
    ├── __init__.py       # must expose root_agent
    └── agent.py          # agent definition and tools

The folder name (weather_agent) is the agent's identifier used by the CLI. The __init__.py must import and expose root_agent — ADK looks for this name specifically when it loads your agent.

Key CLI Commands

All commands are run from my_project/ — the parent directory that contains your agent folder.

# Run the agent interactively in the terminal
adk run weather_agent

# Launch the browser-based chat UI at http://localhost:8000
adk web

# Start a local REST API server for the agent
adk api_server weather_agent
  • adk run — starts an interactive terminal session; type messages, see responses, quit with Ctrl+C.
  • adk web — opens a full chat UI in the browser with a built-in tool-call inspector. Great for demos and debugging.
  • adk api_server — exposes a REST endpoint so you can call the agent from any external application.

3. Building a Basic Agent

An Agent in ADK is defined by four core properties:

  • name — a unique identifier for the agent.
  • model — the Gemini model to use (e.g. gemini-2.0-flash).
  • description — used by orchestrators to decide when to delegate to this agent (important for multi-agent setups).
  • instruction — the system prompt that shapes how the agent behaves.

Let's start with the simplest possible agent — no tools, just a Gemini-powered conversational assistant.

Full runnable code: 01_single_agent.py

weather_agent/agent.py

from google.adk.agents import Agent

root_agent = Agent(
    name="weather_agent",
    model="gemini-2.0-flash",
    description="A helpful assistant that answers weather-related questions.",
    instruction=(
        "You are a friendly weather assistant. "
        "Answer the user's questions clearly and concisely. "
        "If you don't know something, say so honestly."
    ),
)

weather_agent/__init__.py

from . import agent

That's a working agent. At this point it can hold a conversation using Gemini's built-in knowledge, but it can't call external APIs or run custom logic. That's where tools come in.

4. Building a Custom Tool & Integrating It

In ADK a tool is simply a Python function. ADK uses the function's name, type hints, and docstring to automatically build a schema that describes the tool to Gemini — so the model knows when and how to call it. No decorators or wrappers are needed.

Writing the Tool

Let's write a get_weather tool. In a real project this would call a live weather API; here we return mock data to keep the focus on ADK mechanics: 

def get_weather(city: str) -> dict:
    """
    Returns the current weather for the given city.

    Args:
        city: The name of the city to get weather for.

    Returns:
        A dictionary with city, temperature, condition, and humidity.
    """
    # Replace with a real API call in production (e.g. OpenWeatherMap)
    mock_data = {
        "london":   {"temperature": "14°C", "condition": "Cloudy",  "humidity": "78%"},
        "new york": {"temperature": "22°C", "condition": "Sunny",   "humidity": "55%"},
        "tokyo":    {"temperature": "18°C", "condition": "Rainy",   "humidity": "85%"},
    }
    key = city.lower()
    if key in mock_data:
        return {"city": city, **mock_data[key]}
    return {"city": city, "error": f"No weather data found for '{city}'."}

Three rules to keep in mind when writing ADK-compatible tools:

  • Always add type hints — ADK uses them to build the tool schema for the model.
  • Write a clear docstring — Gemini reads this to understand when to call the tool.
  • Return a dict or simple type — ADK serialises the return value back to the model as context.
Integrating the Tool with the Agent

Pass the function directly in the tools list when constructing the agent. Here is the complete agent.py with the tool included:

Full runnable code: 02_custom_tool_agent.py

from google.adk.agents import Agent


def get_weather(city: str) -> dict:
    """
    Returns the current weather for the given city.

    Args:
        city: The name of the city to get weather for.

    Returns:
        A dictionary with city, temperature, condition, and humidity.
    """
    mock_data = {
        "london":   {"temperature": "14°C", "condition": "Cloudy",  "humidity": "78%"},
        "new york": {"temperature": "22°C", "condition": "Sunny",   "humidity": "55%"},
        "tokyo":    {"temperature": "18°C", "condition": "Rainy",   "humidity": "85%"},
    }
    key = city.lower()
    if key in mock_data:
        return {"city": city, **mock_data[key]}
    return {"city": city, "error": f"No weather data found for '{city}'."}


root_agent = Agent(
    name="weather_agent",
    model="gemini-2.0-flash",
    description="A helpful assistant that answers weather-related questions.",
    instruction=(
        "You are a friendly weather assistant. "
        "Use the get_weather tool to look up current conditions for any city the user asks about. "
        "Always present the result in a readable, conversational way."
    ),
    tools=[get_weather],   # wire the tool here
)

When the user asks "What's the weather in London?", Gemini recognises the intent, calls get_weather(city="London") automatically, receives the dict back, and formats a natural-language reply — all without you writing any routing logic.

5. adk run and adk web

adk run — Terminal Chat

From the my_project/ directory run: 

adk run weather_agent

You get an interactive prompt. Type a message and press Enter: 

User: What is the weather like in Tokyo?

Agent: The current weather in Tokyo is rainy with a temperature of 18°C
       and humidity at 85%. Make sure to carry an umbrella!

User: What about New York?

Agent: New York is enjoying sunny skies right now — 22°C with 55% humidity.
       Great day to be outside!

User: exit

Press Ctrl+C or type exit to end the session. History is held in memory and discarded on exit.

adk web — Browser UI

For a more visual experience — especially useful for demos and debugging tool calls: 

adk web

ADK starts a local server:

INFO:     Started server process [12345]
INFO:     Uvicorn running on http://localhost:8000

Open http://localhost:8000 in your browser. The web UI gives you:

  • A dropdown to select any agent in the project
  • A chat window with real-time streaming responses
  • An event timeline that shows every tool call Gemini made, the arguments it passed, and the value your function returned — all collapsible, no extra logging needed

The event timeline is the most useful debugging feature ADK ships with. You can see exactly when the model decided to invoke get_weather, what city it passed, and what your mock data returned — without touching a log file.

6. Conclusion

Google ADK keeps the core concepts intentionally simple so you can focus on agent behaviour rather than plumbing. The things worth remembering:

  • One folder = one agent. The folder name is the agent ID; __init__.py exposes root_agent.
  • Tools are plain Python functions. Type hints and a docstring are all ADK needs to wire them to Gemini.
  • adk run for quick terminal testing; adk web for a visual, debuggable UI.
  • Keep your .env next to the agent folder and never commit it — ADK picks it up automatically.

Start simple, ship a working agent, then layer in complexity — that's the ADK way.
