By the end of Part 2, our customer support bot had fast active-session memory in Redis and could resume cold sessions from a SQL-stored summary. That is solid production infrastructure. But it still has a fundamental blind spot: it does not know who you are.

Every time a user starts a new session, the bot starts from scratch unless you happen to use the same session_id. It cannot connect Monday's billing dispute to Thursday's shipping question, even though both came from the same person. And it calls the LLM even when someone asks a question it has effectively already answered twenty times today.

Part 3 addresses both of these. We add cross-session semantic memory so the bot genuinely knows its users over time, and semantic caching so repeated or similar questions get answered instantly without an LLM call.

1. The Challenge: Knowing Who You Are

Redis sessions are scoped to a session ID. When a user returns after three days with a brand-new session ID, the bot has no way to link the two conversations unless you build that linkage yourself. And even if you do, raw conversation history is a poor format for cross-session recall; it is verbose, unsorted, and full of noise.

What you actually want is for the bot to know things like:

  • This user prefers email updates over SMS.
  • Their last three contacts were all about delayed orders.
  • They were offered a 10% discount last month and declined it.

These are facts, not transcripts. Extracting and storing them efficiently, and retrieving the relevant ones at the start of a new conversation, is exactly what mem0 (for ADK) and LangMem (for LangChain) do.

2. Semantic Matching in Plain English

Classic database lookups are exact: you search for a specific session_id or a keyword. Semantic search is different; it searches by meaning.

The idea is straightforward. Every piece of text (a question, a memory, a sentence) can be converted into a list of numbers called an embedding. Think of it as a fingerprint: texts with similar meaning produce similar-looking fingerprints, even if they use completely different words. “Where is my package?” and “My order hasn't shown up yet” produce fingerprints that are very close together in this numeric space. “What is your return policy?” produces a fingerprint far away from both.

Storing memories as embeddings means you can ask: “which of this user's stored memories are relevant to what they just said?” and get a precise, ranked answer based on meaning rather than keyword overlap.

Want to go deeper? The Building a RAG Pipeline with PDFs, Chroma DB, and Gemma 3 post covers embeddings, vector stores, and similarity search in full detail. The concepts here build directly on that foundation.

3. Semantic Caching

Semantic caching takes the same embedding idea and applies it to LLM responses. When the bot answers “What is your return window?”, that question and its answer are cached as an embedding. The next time any user asks something semantically similar, for example “How many days do I have to return something?”, the cache returns the existing answer without touching the LLM at all.

The benefit is twofold: latency drops from ~800ms to <5ms and LLM API costs drop proportionally. For a support bot handling hundreds of “where is my order?” variations per day, this is significant.

The cache uses a similarity threshold: only return a cached answer if the incoming question is at least X% similar (e.g. 92%). Below that threshold, the LLM is called normally and the new answer is cached.

4. mem0 + Google ADK

mem0 is an open-source library that adds long-term, cross-session memory to AI agents. It automatically extracts key facts from conversations, stores them in a vector database (we use Chroma for consistency with our RAG posts), and retrieves the most relevant ones on demand via semantic search.

Installation & Setup

pip install mem0ai chromadb

import os
from mem0 import Memory

# mem0 config: use Chroma as the local vector store
config = {
    "vector_store": {
        "provider": "chroma",
        "config": {
            "collection_name": "support_memories",
            "path": "./chroma_db",       # local persistent directory
        }
    },
    "llm": {
        "provider": "gemini",
        "config": {
            "model": "gemini-2.0-flash",
            "api_key": os.environ["GEMINI_API_KEY"],
        }
    },
    "embedder": {
        "provider": "gemini",
        "config": {
            "model": "models/text-embedding-004",
            "api_key": os.environ["GEMINI_API_KEY"],
        }
    },
}

memory = Memory.from_config(config)

ADK Agent with mem0 Callbacks

We use ADK's before_agent_callback to inject relevant memories into the session before the LLM is called, and after_agent_callback to extract and save new memories after each turn. 

from google.adk.agents import Agent
from google.adk.runners import Runner
from google.adk.sessions import InMemorySessionService
from google.adk.agents.callback_context import CallbackContext

async def before_agent(callback_context: CallbackContext):
    """Retrieve memories relevant to the user's latest message and
    inject them into session state so the instruction template can use them."""
    user_id      = callback_context.state.get("user_id", "anonymous")
    user_message = callback_context.state.get("last_user_message", "")

    if not user_message:
        return

    # Semantic search: find the top 3 memories closest in meaning to this message
    results = memory.search(query=user_message, user_id=user_id, limit=3)
    memories = results.get("results", [])

    if memories:
        memory_text = "\n".join(
            f"- {m['memory']}" for m in memories
        )
        callback_context.state["user_memories"] = memory_text
    else:
        callback_context.state["user_memories"] = "No prior memories for this user."

async def after_agent(callback_context: CallbackContext):
    """After the agent responds, extract key facts from the conversation
    and save them as memories for future sessions."""
    user_id  = callback_context.state.get("user_id", "anonymous")
    messages = callback_context.state.get("messages", [])

    if messages:
        # mem0 uses the LLM to extract meaningful facts automatically
        memory.add(messages, user_id=user_id)

agent = Agent(
    name="support_bot_v3",
    model="gemini-2.5-flash",
    instruction="""You are a helpful customer support agent for an e-commerce platform.

What you know about this user from past interactions:
{user_memories}

Use this context to personalise your response. If you learn something important
about the user (preferences, recurring issues, resolutions), note it naturally.""",
    before_agent_callback=before_agent,
    after_agent_callback=after_agent,
)

runner = Runner(
    agent=agent,
    app_name="support_app",
    session_service=InMemorySessionService(),
)
How mem0 extracts memories: When you call memory.add(messages, user_id=...), mem0 sends the conversation to the LLM with a dedicated extraction prompt. The LLM identifies facts worth keeping (preferences, issues, resolutions) and returns them as structured memories. These are then embedded and stored in Chroma. Retrieval uses cosine similarity against the stored embeddings.

5. LangMem + LangChain

LangMem is LangChain's answer to the same problem. It integrates natively with LangGraph's memory store and gives your agent two tools: one to save memories and one to search them. The agent decides what is worth remembering and retrieves context proactively; this is called managing memory “in the hot path.”

Installation

pip install langmem langgraph

Agent with memory tools

from langgraph.prebuilt import create_react_agent
from langgraph.store.memory import InMemoryStore
from langmem import create_manage_memory_tool, create_search_memory_tool
from langchain_ollama import ChatOllama

# In-memory store with embedding support
# In production, swap InMemoryStore for AsyncPostgresStore or another persistent backend
store = InMemoryStore(
    index={
        "dims": 768,
        "embed": "ollama:nomic-embed-text",   # local embedding model via Ollama
    }
)

llm = ChatOllama(model="llama3.2")

# The namespace "{user_id}" is filled at runtime, keeping memories per-user
agent = create_react_agent(
    llm,
    tools=[
        # manage_memory: agent can save, update, or delete memories mid-conversation
        create_manage_memory_tool(namespace=("support_memories", "{user_id}")),
        # search_memory: agent can look up past context before answering
        create_search_memory_tool(namespace=("support_memories", "{user_id}")),
    ],
    store=store,
    state_modifier="""You are a helpful customer support agent for an e-commerce platform.

Before answering any question, search your memory for relevant context about this user.
After resolving an issue, save important facts (preferences, issue type, resolution, order IDs)
to memory so you can personalise future interactions.""",
)

# The user_id flows through the namespace automatically
response = agent.invoke(
    {"messages": [{"role": "user", "content": "My order #1234 still hasn't arrived."}]},
    config={"configurable": {"user_id": "user_42"}},
)
print(response["messages"][-1].content)

# Next session - different session_id, same user_id
# The agent will find memories from the previous interaction via search_memory_tool
response2 = agent.invoke(
    {"messages": [{"role": "user", "content": "Hi, I have another issue with my order."}]},
    config={"configurable": {"user_id": "user_42"}},
)
print(response2["messages"][-1].content)
# Agent might respond: "Hi! I see you had an issue with order #1234 last time. Is this a new order?"
LangMem background processing: LangMem also supports a background memory manager that processes conversations after they complete, extracting, consolidating, and deduplicating memories without adding latency to the live conversation. This is useful for high-volume deployments where you do not want memory extraction to slow down each turn.

6. Semantic Caching with LangCache (ADK)

For the ADK path, Redis's LangCache provides managed semantic caching. It intercepts every LLM call, checks whether a semantically similar question has been answered before, and returns the cached answer if the similarity exceeds the threshold. Embedding computation happens server-side so there is no extra work in your Python code. 

pip install adk-redis[langcache]

from adk_redis.caching import LangCacheSemanticCacheProvider
from google.adk.agents import Agent
from google.adk.runners import Runner
from google.adk.sessions import InMemorySessionService

# LangCache connects to the Agent Memory Server for server-side embedding
cache_provider = LangCacheSemanticCacheProvider(
    api_base_url="http://localhost:8088",
    namespace="support_cache",
    similarity_threshold=0.92,   # 0.92 = only return cache hits for very similar questions
)

agent = Agent(
    name="support_bot_cached",
    model="gemini-2.5-flash",
    instruction="You are a helpful customer support agent.",
)

runner = Runner(
    agent=agent,
    app_name="support_app",
    session_service=InMemorySessionService(),
    cache_provider=cache_provider,   # LangCache wraps all LLM calls transparently
)

# First call: cache miss - LLM is invoked, response stored
# "What is your return policy?" → LLM answers, cached

# Second call (different phrasing): cache hit - no LLM call
# "How many days do I have to return an item?" → semantically close → cached answer returned instantly
The similarity_threshold controls how strict the cache is. At 0.92, only near-identical questions hit the cache. Lowering it to 0.80 allows fuzzier matches but risks returning answers that are slightly off-topic. Tune this based on your domain; support FAQs can tolerate a lower threshold than medical or legal advice bots.

7. The Full Async Pipeline

All three layers can run asynchronously in a production setup. The ADK path is already async-native; callbacks are async def functions and the Runner uses an event loop. mem0 supports async clients. LangMem's create_react_agent and LangGraph's store are async-compatible. 

import asyncio
from mem0 import AsyncMemory   # async client

async_memory = AsyncMemory.from_config(config)

async def before_agent(callback_context: CallbackContext):
    user_id      = callback_context.state.get("user_id", "anonymous")
    user_message = callback_context.state.get("last_user_message", "")
    if not user_message:
        return

    # Non-blocking: other users' requests proceed while this awaits
    results = await async_memory.search(query=user_message, user_id=user_id, limit=3)
    memories = results.get("results", [])
    memory_text = "\n".join(f"- {m['memory']}" for m in memories) if memories else ""
    callback_context.state["user_memories"] = memory_text or "No prior memories."

async def after_agent(callback_context: CallbackContext):
    user_id  = callback_context.state.get("user_id", "anonymous")
    messages = callback_context.state.get("messages", [])
    if messages:
        # Fire-and-forget: don't block the response while memories are extracted
        asyncio.create_task(async_memory.add(messages, user_id=user_id))
Want to understand fire-and-forget tasks? The Python Concurrency Part 1: asyncio, Coroutines, Tasks and Synchronisation post covers create_task vs await and when to use each. The Python Concurrency Project Part 4 shows a full multi-agent async pipeline that mirrors the architecture here.

9. User Experience Impact

The technical improvements translate directly into observable user experience changes. Consider the same user contacting support three times over a month:

With Part 1 (SQL only)

  • Contact 1: Bot has no context. User explains their account, order number, and problem from scratch.
  • Contact 2 (new session): Bot asks the same qualifying questions again. User is frustrated.
  • Contact 3: Same story. User considers not using the service again.

With Part 2 (Redis + SQL)

  • Contact 1: Normal conversation stored and compressed.
  • Contact 2 (returning): If within the same session lifecycle, context is restored. If a new session, bot loads the summary and can reference the previous issue.
  • Contact 3: Consistent, but summary may only capture the most recent session.

With Part 3 (mem0 / LangMem)

  • Contact 1: Conversation is indexed. Key facts extracted: “prefers email updates,” “order #1234,” “delayed delivery resolved with reship.”
  • Contact 2: Bot retrieves these facts. Says: “Welcome back! I see your previous order was delayed. I want to make sure this goes smoother. Let me check your new order right away.”
  • Contact 3: Bot knows the user's history, frustration level, and past resolutions. Personalised from turn 1. Latency is also lower because similar FAQ questions hit the semantic cache.

Summary

  • Semantic matching converts text into numeric fingerprints (embeddings). Similar meaning → similar fingerprints → semantic retrieval.
  • mem0 (ADK path): automatically extracts facts from conversations, stores them in Chroma, retrieves the relevant ones via semantic search at the start of each session. Configure via before_agent_callback + after_agent_callback.
  • LangMem (LangChain path): gives the agent two tools, create_manage_memory_tool and create_search_memory_tool, so it manages its own memory in the hot path. Namespace scoping (("{user_id}",)) keeps memories per-user.
  • Semantic caching (LangCache via adk-redis[langcache]): similar questions get the same cached answer without an LLM call. Sub-5ms responses for FAQ-style traffic, significant cost savings.
  • Use AsyncMemory and asyncio.create_task to run memory extraction as a fire-and-forget background task so it never adds latency to the user-facing response.
  • The full journey: SQLite for dev, Redis + SQL for production sessions, mem0 / LangMem + semantic cache for production personalisation at scale.
