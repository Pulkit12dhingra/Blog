In Part 1 we gave our customer support bot a memory using SQLite. It can now remember the current conversation and load it back after a server restart. But there are two problems we still have not solved: speed and session continuity.

This part upgrades the architecture to a two-layer memory system: Redis handles the fast, hot path for active conversations, and SQL acts as the permanent archive. We introduce rolling windows and running summaries so the bot never loses context, even when a user returns after days of inactivity.

1. The Problem with SQL Alone

SQLite reads from disk. Disk is slow relative to RAM; typically 5–50ms per query vs 0.1–1ms for an in-memory store. For a chat application where the bot needs to load conversation history on every single turn, those milliseconds compound fast.

There is a second problem: growing context. As a session gets longer, loading the entire message history means sending more and more text to the LLM. Token counts grow, costs rise, and you eventually hit the context limit. We need a smarter approach to what we keep hot and what we compress.

2. Redis in One Paragraph

Redis is an in-memory data store. Think of it as your computer's RAM organised as a database: everything lives in memory so reads and writes are measured in microseconds, not milliseconds. It supports key-value pairs, lists, hashes, and more. Critically, every key can have a TTL (time-to-live); you tell Redis "delete this key after 2 hours of inactivity" and it handles eviction automatically. This makes Redis a natural fit for active session storage: sessions that go quiet get cleaned up without any cron jobs.

3. The Two-Layer Architecture

The new system uses two complementary stores with clearly defined responsibilities:

LayerStoreWhat lives hereLifetime
Working memory Redis Last N turns (rolling window) + running summary of the whole session TTL-based (e.g. 2 hours of inactivity)
Long-term archive SQL (SQLite / PostgreSQL) Full conversation summary written on session close Permanent (until you explicitly delete)

Rolling Window

Instead of storing every message forever in Redis, we keep only the last N turns (say, 10). When turn 11 arrives, turn 1 is dropped from Redis (but not deleted; it has already been archived to SQL). This keeps the active context lean and token-efficient.

In raw Redis this is a list with LPUSH (push new message to the front) + LTRIM (trim to max N items). The adk-redis library does this automatically via the Agent Memory Server.

Running Summary

Alongside the rolling window, a second Redis key stores a compressed plain-English summary of everything discussed so far, updated after each turn. Even though turn 1 has been evicted from the window, its key facts (order number, problem description, resolution attempted) live on in the summary. The LLM receives both the window (recent verbatim turns) and the summary (older compressed context). This is the pattern used by most production chatbot systems.

Cold Resume

When a session TTL expires in Redis (user was inactive for hours), the next time they return:

  1. The bot checks Redis: key is gone (miss).
  2. It loads the session summary from SQL (fast single-row lookup).
  3. It seeds Redis with that summary as the starting context.
  4. The conversation resumes naturally. The bot can say “Welcome back. Last time we were working on your order #1234...”

4. Setting Up adk-redis

adk-redis is the official Redis integration library for Google ADK. It connects ADK's session and memory interfaces to Redis Agent Memory Server, a purpose-built service that handles working memory, auto-summarisation, and long-term memory storage. See the official Redis ADK docs for full reference.

Docker Setup (Redis 8.4 + Agent Memory Server)

# 1. Start Redis 8.4 with vector search support
docker run -d --name redis -p 6379:6379 redis:8.4-alpine

# 2. Start Agent Memory Server (handles summarisation, working memory, long-term memory)
docker run -d --name agent-memory-server -p 8088:8088 \
  -e REDIS_URL=redis://host.docker.internal:6379 \
  -e GEMINI_API_KEY=your-gemini-key \
  -e GENERATION_MODEL=gemini/gemini-2.0-flash \
  -e EMBEDDING_MODEL=gemini/text-embedding-004 \
  redislabs/agent-memory-server:latest \
  agent-memory api --host 0.0.0.0 --port 8088 --task-backend=asyncio

# Install the ADK Redis integration
pip install adk-redis[memory]
The Agent Memory Server is the key piece: it talks directly to Redis and exposes a REST API on port 8088 that adk-redis calls. It handles the rolling window and auto-summarisation natively; you do not need to implement these yourself when using ADK.

5. ADK + adk-redis Implementation

from google.adk.agents import Agent
from google.adk.runners import Runner
from google.adk.agents.callback_context import CallbackContext
from adk_redis.sessions import (
    RedisWorkingMemorySessionService,
    RedisWorkingMemorySessionServiceConfig,
)

# Point adk-redis at the Agent Memory Server
session_service = RedisWorkingMemorySessionService(
    config=RedisWorkingMemorySessionServiceConfig(
        api_base_url="http://localhost:8088",
        default_namespace="support_app",
    )
)

async def after_agent_turn(callback_context: CallbackContext):
    """After every agent turn, persist the session to long-term memory.

    The Agent Memory Server automatically manages:
    - The rolling window (last N messages in Redis)
    - The running summary (incrementally updated after each call)
    - Archival to its own long-term store (backed by Redis)
    """
    await callback_context.add_session_to_memory()

agent = Agent(
    name="support_bot_v2",
    model="gemini-2.5-flash",
    instruction="""You are a helpful customer support agent for an e-commerce platform.
You have access to a summary of previous interactions. Use it to personalise your response.""",
    after_agent_callback=after_agent_turn,
)

runner = Runner(
    agent=agent,
    app_name="support_app",
    session_service=session_service,  # Redis-backed, replaces InMemorySessionService
)

The key difference from Part 1 is the session_service. Swapping InMemorySessionService for RedisWorkingMemorySessionService is all it takes for ADK. The rolling window and summary management are handled server-side by the Agent Memory Server.

6. LangChain + redis-py: Rolling Window + Summary + Cold Resume

The LangChain path uses redis-py directly, giving you full control over the rolling window and summary keys. We also add the SQL cold-resume pattern explicitly. 

pip install redis langchain-community

Redis helpers: rolling window + summary key

import redis
import sqlite3
import json

r = redis.Redis(host="localhost", port=6379, decode_responses=True)

WINDOW_SIZE = 10       # keep last 10 turns in Redis
SESSION_TTL  = 7200    # 2 hours of inactivity before Redis evicts the session
SUMMARY_TTL  = 86400   # summary key lives 24 hours (longer than the window)

# ── Rolling window ──────────────────────────────────────────────────

def push_message(session_id: str, role: str, content: str):
    """Add a message to the rolling window and trim to WINDOW_SIZE."""
    key = f"window:{session_id}"
    message = json.dumps({"role": role, "content": content})
    r.lpush(key, message)           # push to front
    r.ltrim(key, 0, WINDOW_SIZE - 1)  # keep only the last WINDOW_SIZE items
    r.expire(key, SESSION_TTL)

def get_window(session_id: str) -> list[dict]:
    """Return the last N messages in chronological order."""
    key = f"window:{session_id}"
    raw = r.lrange(key, 0, -1)
    return [json.loads(m) for m in reversed(raw)]  # reverse: LPUSH stores newest first

# ── Running summary ─────────────────────────────────────────────────

def update_summary(session_id: str, new_line: str):
    """Append a one-line summary of the latest turn."""
    key = f"summary:{session_id}"
    existing = r.get(key) or ""
    r.set(key, existing + f"\n- {new_line}", ex=SUMMARY_TTL)

def get_summary(session_id: str) -> str:
    return r.get(f"summary:{session_id}") or ""

# ── SQL long-term store (cold-resume) ───────────────────────────────

def archive_summary(session_id: str):
    """Write the running summary to SQL on session close."""
    summary = get_summary(session_id)
    if not summary:
        return
    conn = sqlite3.connect("support.db")
    conn.execute("""
        CREATE TABLE IF NOT EXISTS session_summaries (
            session_id TEXT PRIMARY KEY,
            summary    TEXT,
            updated_at REAL DEFAULT (unixepoch())
        )""")
    conn.execute(
        "INSERT OR REPLACE INTO session_summaries (session_id, summary) VALUES (?,?)",
        (session_id, summary)
    )
    conn.commit()
    conn.close()

def load_summary_from_sql(session_id: str) -> str:
    """Load the archived summary when Redis has no entry (cold-resume)."""
    conn = sqlite3.connect("support.db")
    row = conn.execute(
        "SELECT summary FROM session_summaries WHERE session_id=?",
        (session_id,)
    ).fetchone()
    conn.close()
    return row[0] if row else ""

def restore_session(session_id: str):
    """On cold-resume: load summary from SQL and seed Redis."""
    if not r.exists(f"summary:{session_id}"):
        archived = load_summary_from_sql(session_id)
        if archived:
            r.set(f"summary:{session_id}", archived, ex=SUMMARY_TTL)

LangChain chain with window + summary context

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.messages import HumanMessage, AIMessage
from langchain_ollama import ChatOllama

llm = ChatOllama(model="llama3.2")

def chat(session_id: str, user_input: str) -> str:
    # Cold-resume: restore summary from SQL if Redis has expired
    restore_session(session_id)

    summary = get_summary(session_id)
    window  = get_window(session_id)

    # Build message list for the LLM: system + past window + new user turn
    messages = [
        ("system",
         f"You are a helpful customer support agent.\n\n"
         f"Summary of past interactions:\n{summary or 'No prior context.'}")
    ]
    for msg in window:
        if msg["role"] == "user":
            messages.append(("human", msg["content"]))
        else:
            messages.append(("ai", msg["content"]))
    messages.append(("human", user_input))

    prompt = ChatPromptTemplate.from_messages(messages)
    response = (prompt | llm).invoke({})
    response_text = response.content

    # Update Redis
    push_message(session_id, "user", user_input)
    push_message(session_id, "model", response_text)
    update_summary(session_id, f"User: {user_input[:60]} | Bot: {response_text[:60]}")

    return response_text

7. Async Upgrade

Redis is a network service, which means every call involves a round-trip over the loopback. Under concurrent load, sync Redis calls block the thread just like sync SQLite. aioredis gives you async-native Redis clients.

The ADK path is already fully async; after_agent_callback is async def and the Runner uses an async event loop. For the LangChain path: 

pip install redis[asyncio]   # aioredis is now part of the redis package

import redis.asyncio as aioredis

# Create once at startup, reuse across requests
redis_client = aioredis.from_url("redis://localhost:6379", decode_responses=True)

async def push_message_async(session_id: str, role: str, content: str):
    key = f"window:{session_id}"
    message = json.dumps({"role": role, "content": content})
    async with redis_client.pipeline(transaction=True) as pipe:
        await pipe.lpush(key, message)
        await pipe.ltrim(key, 0, WINDOW_SIZE - 1)
        await pipe.expire(key, SESSION_TTL)
        await pipe.execute()

async def get_window_async(session_id: str) -> list[dict]:
    key = f"window:{session_id}"
    raw = await redis_client.lrange(key, 0, -1)
    return [json.loads(m) for m in reversed(raw)]

async def restore_session_async(session_id: str):
    exists = await redis_client.exists(f"summary:{session_id}")
    if not exists:
        archived = await load_summary_from_sql_async(session_id)
        if archived:
            await redis_client.set(f"summary:{session_id}", archived, ex=SUMMARY_TTL)
Threading vs async for Redis: The async approach is ideal when your web framework is async (FastAPI, aiohttp). If you are using a sync framework (Flask, Django without ASGI) and want concurrency, use a thread pool with the sync redis-py client instead. See Python Concurrency Part 3: Threading, ThreadPoolExecutor & the GIL for the detailed comparison.

9. What Is Still Missing

The bot now handles fast active sessions, graceful eviction, and smooth cold-resume. But there is one thing it still cannot do: recognise the same user across completely separate conversations.

If user 42 contacts support today about a billing issue, then comes back next month with a shipping question, the bot starts from a blank slate (unless you manually link the session IDs). It does not know that user 42 had a billing dispute last month and resolved it with a refund. It cannot say “I see you had an issue with us before; let me make sure we handle this quickly.”

For that, we need cross-session semantic memory: the ability to extract key facts from conversations, store them as embeddings, and retrieve them by meaning, not just by session ID. That is what Part 3 is about.

Summary

  • Redis keeps active session context in RAM for sub-millisecond reads, far faster than SQLite on disk.
  • A rolling window (last N turns) keeps token counts fixed and predictable.
  • A running summary alongside the window preserves older context without storing every message verbatim.
  • SQL acts as the permanent archive. On cold-resume, a single summary row is loaded and Redis is reseeded; the conversation picks up naturally.
  • ADK path: swap InMemorySessionService for RedisWorkingMemorySessionService (adk-redis) and add add_session_to_memory() in your after-agent callback. The Agent Memory Server handles rolling window and summarisation automatically.
  • LangChain path: use redis-py with LPUSH + LTRIM for the window, a separate summary key, and aiosqlite for SQL archival and cold-resume.
  • Upgrade to async Redis (redis[asyncio]) to serve many concurrent users without thread blocking.
  • Continue to Part 3 to add cross-session semantic memory and semantic caching.
