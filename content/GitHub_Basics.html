üìö What is Git and GitHub?

Git: The Version Control System

Git is a distributed version control system (VCS) that tracks changes in your code over time. Think of it as a time machine for your projects ‚Äî you can save snapshots of your work, experiment without fear, and collaborate seamlessly with others.

Why Git matters:

  • Track changes: See what changed, when, and by whom
  • Collaborate safely: Multiple people can work on the same project without conflicts
  • Experiment freely: Create branches to test new features without breaking your main code
  • Backup automatically: Your entire project history is distributed across all contributors
  • Undo mistakes: Roll back to any previous state with a single command
GitHub: Git Hosting + Social Coding

GitHub is a cloud-based platform that hosts Git repositories and adds powerful collaboration features. It's like Git plus a social network for developers.

GitHub provides:

  • Remote hosting: Store your repositories in the cloud
  • Collaboration tools: Pull requests, code reviews, issues, and project boards
  • CI/CD: GitHub Actions for automated testing and deployment
  • Documentation: README files, wikis, and GitHub Pages for static sites
  • Community: Discover, fork, and contribute to millions of open source projects
üîë Key Difference: Git is the tool, GitHub is the platform. You use Git locally on your machine, and GitHub as a remote hub for collaboration.

üåø Understanding Branches, Commits, and Pull Requests

Commits: Saving Snapshots

A commit is like a checkpoint in a video game ‚Äî it saves the current state of your project with a message describing what changed. 

# Make changes to your files
git add file1.py file2.py       # Stage specific files
git add .                        # Stage all changes

# Create a commit with a descriptive message
git commit -m "Add user authentication feature"

# View commit history
git log
git log --oneline --graph        # Compact, visual format

Good commit practices:

  • Write clear, descriptive messages (not "fix bug" but "Fix login validation for empty passwords")
  • Make small, focused commits (one feature/fix per commit)
  • Commit often to create a detailed history
  • Use present tense: "Add feature" not "Added feature"
Branches: Parallel Development

Branches let you diverge from the main codebase to work on features, fixes, or experiments without affecting the stable version. 

# Create a new branch
git branch feature/user-profile

# Switch to that branch
git checkout feature/user-profile

# Or do both in one command
git checkout -b feature/user-profile

# List all branches
git branch                       # Local branches
git branch -r                    # Remote branches
git branch -a                    # All branches

# Switch back to main
git checkout main

# Delete a branch
git branch -d feature/user-profile

Common branch naming conventions:

  • main or master: The production-ready code
  • develop: Integration branch for features
  • feature/feature-name: New features
  • bugfix/bug-description: Bug fixes
  • hotfix/critical-issue: Urgent production fixes
Pull Requests (PRs): Code Review & Merging

A pull request (PR) is how you propose changes to a repository. It's a request to merge your branch into another branch (usually main).

Pull request workflow:

  1. Create a branch and make changes
  2. Push your branch to GitHub
  3. Open a PR on GitHub's web interface
  4. Team members review your code, leave comments
  5. You address feedback and push more commits
  6. Once approved, the PR is merged
# After making commits on your branch
git push origin feature/user-profile

# Then on GitHub:
# 1. Go to your repository
# 2. Click "Pull requests" ‚Üí "New pull request"
# 3. Select your branch
# 4. Add title and description
# 5. Request reviewers
# 6. Click "Create pull request"

üìã The Staging Area and Git CLI

The Three-State Architecture

Git has three main states your files can be in:

  1. Working Directory: Where you edit files
  2. Staging Area (Index): Where you prepare commits
  3. Repository (.git): Where commits are stored permanently
# Check status of files
git status

# Output shows:
# - Untracked files (never added to Git)
# - Modified files (changed since last commit)
# - Staged files (ready to be committed)
Staging Area Commands
# Add specific files to staging
git add file1.py file2.py

# Add all changes
git add .

# Add all Python files
git add *.py

# Unstage a file (keep changes in working directory)
git reset file1.py

# Unstage all files
git reset

# Discard changes in working directory (careful!)
git checkout -- file1.py

# See what you're about to commit
git diff --staged
Essential Git CLI Commands

Setup & Configuration:

# Configure your identity
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# Set default branch name
git config --global init.defaultBranch main

# View all settings
git config --list

Repository Management:

# Initialize a new repository
git init

# Clone an existing repository
git clone https://github.com/username/repo.git

# Add a remote repository
git remote add origin https://github.com/username/repo.git

# View remotes
git remote -v

# Change remote URL
git remote set-url origin https://github.com/username/new-repo.git

Working with Changes:

# See what changed
git status                       # Summary of changes
git diff                         # Detailed changes (unstaged)
git diff --staged                # Changes in staging area
git diff branch1..branch2        # Compare branches

# Commit changes
git commit -m "Message"          # Commit staged files
git commit -am "Message"         # Stage and commit tracked files
git commit --amend               # Modify last commit

# View history
git log                          # Full commit history
git log --oneline                # Compact view
git log --graph --all            # Visual branch history
git log -p                       # Show diffs in each commit
git log --author="Name"          # Filter by author

Syncing with Remote:

# Fetch changes from remote (doesn't merge)
git fetch origin

# Pull changes (fetch + merge)
git pull origin main

# Push your changes
git push origin main
git push origin feature/my-branch

# Push and set upstream
git push -u origin main          # First push

# Force push (use with caution!)
git push --force origin main

Undoing Changes:

# Undo last commit (keep changes)
git reset --soft HEAD~1

# Undo last commit (discard changes)
git reset --hard HEAD~1

# Revert a commit (creates new commit)
git revert abc123

# Discard all local changes
git reset --hard HEAD

# Remove untracked files
git clean -fd

üîÑ Common Git Workflows

Basic Feature Development Flow
# 1. Update main branch
git checkout main
git pull origin main

# 2. Create feature branch
git checkout -b feature/new-feature

# 3. Make changes and commit
git add .
git commit -m "Implement new feature"

# 4. Push to remote
git push origin feature/new-feature

# 5. Open PR on GitHub
# 6. After approval, merge via GitHub UI
# 7. Clean up
git checkout main
git pull origin main
git branch -d feature/new-feature
Syncing Fork with Original Repository
# Add original repo as upstream
git remote add upstream https://github.com/original/repo.git

# Fetch from upstream
git fetch upstream

# Merge upstream changes
git checkout main
git merge upstream/main

# Push to your fork
git push origin main

üí° Pro Tips and Best Practices

  • Commit often: Small, frequent commits are better than large, infrequent ones
  • Write meaningful commit messages: Your future self will thank you
  • Never commit secrets: API keys, passwords, and tokens don't belong in version control
  • Use .gitignore: Exclude build artifacts, dependencies, and OS files (covered in the advanced blog)
  • Pull before push: Always sync with remote before pushing to avoid conflicts
  • Branch for everything: Never work directly on main
  • Review before commit: Use git diff to verify what you're committing
  • Learn to resolve conflicts: They're inevitable in team projects
