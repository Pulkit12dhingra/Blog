What is Git and GitHub?

Git: The Version Control System

Git is a distributed version control system (VCS) that tracks changes in your code over time. Think of it as a time machine for your projects ‚Äî you can save snapshots of your work, experiment without fear, and collaborate seamlessly with others.

Why Git matters:

  • Track changes: See what changed, when, and by whom
  • Collaborate safely: Multiple people can work on the same project without conflicts
  • Experiment freely: Create branches to test new features without breaking your main code
  • Backup automatically: Your entire project history is distributed across all contributors
  • Undo mistakes: Roll back to any previous state with a single command
GitHub: Git Hosting + Social Coding

GitHub is a cloud-based platform that hosts Git repositories and adds powerful collaboration features. It's like Git plus a social network for developers.

GitHub provides:

  • Remote hosting: Store your repositories in the cloud
  • Collaboration tools: Pull requests, code reviews, issues, and project boards
  • CI/CD: GitHub Actions for automated testing and deployment
  • Documentation: README files, wikis, and GitHub Pages for static sites
  • Community: Discover, fork, and contribute to millions of open source projects
Key Difference: Git is the tool, GitHub is the platform. You use Git locally on your machine, and GitHub as a remote hub for collaboration.

Understanding Branches, Commits, and Pull Requests

Commits: Saving Snapshots

A commit is like a checkpoint in a video game ‚Äî it saves the current state of your project with a message describing what changed. Think of it as taking a snapshot of your entire project at a specific moment in time.

What happens when you commit?

  • Git creates a unique ID (SHA hash): Every commit gets a 40-character identifier (e.g., a3f2b1c...)
  • Stores metadata: Author name, email, timestamp, and commit message
  • Records changes: Git saves what changed (diffs), not the entire file again
  • Creates a timeline: Each commit points to its parent, forming a chain of history
  • Becomes permanent: Once committed, you can always go back to this exact state

The anatomy of a commit: Each commit contains three key pieces of information:

  1. What changed: The actual code modifications (diff)
  2. When and by whom: Timestamp and author information
  3. Why it changed: Your commit message explaining the purpose
Think of commits as save points: You can travel back to any commit to see what your project looked like at that moment, or even restore your project to that state if needed. 
# Make changes to your files
git add file1.py file2.py       # Stage specific files
git add .                        # Stage all changes

# Create a commit with a descriptive message
git commit -m "Add user authentication feature"

# View commit history
git log
git log --oneline --graph        # Compact, visual format

Good commit practices:

  • Write clear, descriptive messages (not "fix bug" but "Fix login validation for empty passwords")
  • Make small, focused commits (one feature/fix per commit)
  • Commit often to create a detailed history
  • Use present tense: "Add feature" not "Added feature"
Branches: Parallel Development

Branches are one of Git's most powerful features. They let you diverge from the main codebase to work on features, fixes, or experiments without affecting the stable version. Imagine branches as parallel universes where you can make changes safely.

Why branches are essential:

  • Isolated development: Work on new features without breaking the main codebase
  • Experimentation: Try risky changes without fear ‚Äî if it fails, just delete the branch
  • Collaboration: Multiple developers can work on different features simultaneously
  • Code review: Changes can be reviewed before merging into the main branch
  • Version management: Maintain different versions (production, staging, development) as separate branches

How branches work: When you create a branch, Git creates a pointer to the current commit. As you make new commits on that branch, the pointer moves forward, but the main branch stays where it was. Later, you can merge these changes back into main.

Branch visualization: Imagine a tree ‚Äî the trunk is your main branch, and each branch is a limb growing in its own direction. Eventually, branches can merge back into the trunk.

Branch lifecycle:

  1. Create: Start a new branch from an existing commit (usually from main)
  2. Develop: Make commits on your branch as you build your feature
  3. Push: Share your branch with others by pushing to remote
  4. Review: Open a pull request for team review
  5. Merge: Combine your changes back into the main branch
  6. Delete: Clean up the branch once it's merged
# Create a new branch
git branch feature/user-profile

# Switch to that branch
git checkout feature/user-profile

# Or do both in one command
git checkout -b feature/user-profile

# List all branches
git branch                       # Local branches
git branch -r                    # Remote branches
git branch -a                    # All branches

# Switch back to main
git checkout main

# Delete a branch
git branch -d feature/user-profile

Common branch naming conventions:

  • main or master: The production-ready code
  • develop: Integration branch for features
  • feature/feature-name: New features
  • bugfix/bug-description: Bug fixes
  • hotfix/critical-issue: Urgent production fixes
Pull Requests (PRs): Code Review & Merging

A pull request (PR) is GitHub's mechanism for proposing, reviewing, and discussing code changes before they're merged into the main codebase. Despite the name, you're not "pulling" anything ‚Äî you're asking maintainers to "pull" your changes into their repository.

Why pull requests matter:

  • Code quality: Get feedback from experienced developers before code goes live
  • Knowledge sharing: Team members learn from reviewing each other's code
  • Bug prevention: Multiple eyes catch issues you might have missed
  • Documentation: PR discussions create a record of why decisions were made
  • Automated checks: CI/CD runs tests automatically on every PR
  • Safe merging: Ensures only reviewed, tested code enters production

Anatomy of a pull request:

  • Title: Clear, concise summary (e.g., "Add user authentication with JWT")
  • Description: Detailed explanation of what changed and why
  • Changes tab: Shows the actual code diff (what was added/removed)
  • Commits tab: Lists all commits included in this PR
  • Checks: Status of automated tests and builds
  • Reviewers: People assigned to review your code
  • Comments: Discussion thread for feedback and questions
The PR workflow cycle: PR created ‚Üí Automated tests run ‚Üí Reviewers comment ‚Üí Author fixes issues ‚Üí Tests run again ‚Üí Reviewers approve ‚Üí PR merged ‚Üí Branch deleted.

Best practices for pull requests:

  • Keep them small: Smaller PRs are easier to review (aim for <400 lines changed)
  • One purpose per PR: Don't mix feature work with unrelated refactoring
  • Write good descriptions: Explain the "why" not just the "what"
  • Link issues: Reference related issues with Fixes #123 or Closes #456
  • Add screenshots: Visual changes should include before/after images
  • Respond promptly: Address reviewer feedback quickly to keep momentum
  • Be respectful: Reviews are about code quality, not personal criticism

Pull request workflow:

  1. Create a branch and make changes
  2. Push your branch to GitHub
  3. Open a PR on GitHub's web interface
  4. Team members review your code, leave comments
  5. You address feedback and push more commits
  6. Once approved, the PR is merged
# After making commits on your branch
git push origin feature/user-profile

# Then on GitHub:
# 1. Go to your repository
# 2. Click "Pull requests" ‚Üí "New pull request"
# 3. Select your branch
# 4. Add title and description
# 5. Request reviewers
# 6. Click "Create pull request"

The Staging Area and Git CLI

The Three-State Architecture

Understanding Git's three-state architecture is crucial to mastering version control. Git operates differently from older systems like SVN ‚Äî it has an intermediate step between editing files and committing them. This is the staging area (also called the "index").

The three states explained:

  1. Working Directory (Modified):
    • This is where you actively edit your files
    • Changes here are not tracked yet ‚Äî they only exist on your local filesystem
    • If you delete a file here without staging/committing, it's gone forever
    • Think of this as your "draft" workspace
  2. Staging Area / Index (Staged):
    • An intermediate area where you prepare your next commit
    • You selectively choose which changes to include in the next commit
    • Allows you to commit only part of your changes (e.g., commit bugfix but not unfinished feature)
    • Think of this as a "shopping cart" where you collect changes before checking out
  3. Repository / .git directory (Committed):
    • Permanent storage of your project's history
    • Once committed, your changes are safe and recoverable
    • Each commit is identified by a unique SHA hash
    • Think of this as your "published" or "saved" history
Why staging matters: The staging area gives you fine-grained control. You can work on multiple features simultaneously, but commit them separately with clear, focused commit messages. This keeps your history clean and makes it easier to review or revert specific changes later.

File state transitions: 

Working Directory  ‚Üí  Staging Area  ‚Üí  Repository
   (modified)      git add     (staged)   git commit  (committed)
                              ‚Üê  git reset
                   git checkout --  ‚Üê
# Check status of files
git status

# Output shows:
# - Untracked files (never added to Git)
# - Modified files (changed since last commit)
# - Staged files (ready to be committed)
Staging Area Commands

The staging area is where you build your next commit piece by piece. These commands help you control exactly what goes into each commit, giving you the flexibility to create clean, logical commits even when you've made many changes.

Common staging scenarios:

  • Selective staging: You fixed two bugs but want separate commits for clarity
  • Partial commits: You're working on a feature but want to commit a quick documentation fix first
  • Review before commit: Stage files, review the diff, then commit only if satisfied
  • Unstaging mistakes: You accidentally staged a file with sensitive data
# Add specific files to staging
git add file1.py file2.py

# Add all changes
git add .

# Add all Python files
git add *.py

# Unstage a file (keep changes in working directory)
git reset file1.py

# Unstage all files
git reset

# Discard changes in working directory (careful!)
git checkout -- file1.py

# See what you're about to commit
git diff --staged
Essential Git CLI Commands

The Git command-line interface (CLI) is powerful and versatile. While GUI tools exist, understanding the CLI gives you complete control and helps you troubleshoot issues. Here's a comprehensive guide to the most important commands, organized by task.

Pro tip: Don't try to memorize everything at once! Start with the basics (status, add, commit, push, pull) and gradually expand your toolkit as you encounter new scenarios.

Setup & Configuration:

These commands configure Git on your machine. You typically run them once after installing Git:

# Configure your identity
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# Set default branch name
git config --global init.defaultBranch main

# View all settings
git config --list

Repository Management:

These commands help you start new projects or connect to existing ones:

# Initialize a new repository
git init

# Clone an existing repository
git clone https://github.com/username/repo.git

# Add a remote repository
git remote add origin https://github.com/username/repo.git

# View remotes
git remote -v

# Change remote URL
git remote set-url origin https://github.com/username/new-repo.git

Working with Changes:

# See what changed
git status                       # Summary of changes
git diff                         # Detailed changes (unstaged)
git diff --staged                # Changes in staging area
git diff branch1..branch2        # Compare branches

# Commit changes
git commit -m "Message"          # Commit staged files
git commit -am "Message"         # Stage and commit tracked files
git commit --amend               # Modify last commit

# View history
git log                          # Full commit history
git log --oneline                # Compact view
git log --graph --all            # Visual branch history
git log -p                       # Show diffs in each commit
git log --author="Name"          # Filter by author

Syncing with Remote:

These commands synchronize your local repository with GitHub (or other remote hosts). Understanding the difference between fetch, pull, and push is crucial:

  • fetch: Downloads changes from remote but doesn't modify your working files (safe to run anytime)
  • pull: Downloads changes AND merges them into your current branch (fetch + merge)
  • push: Uploads your local commits to the remote repository
# Fetch changes from remote (doesn't merge)
git fetch origin

# Pull changes (fetch + merge)
git pull origin main

# Push your changes
git push origin main
git push origin feature/my-branch

# Push and set upstream
git push -u origin main          # First push

# Force push (use with caution!)
git push --force origin main

Undoing Changes:

# Undo last commit (keep changes)
git reset --soft HEAD~1

# Undo last commit (discard changes)
git reset --hard HEAD~1

# Revert a commit (creates new commit)
git revert abc123

# Discard all local changes
git reset --hard HEAD

# Remove untracked files
git clean -fd

Common Git Workflows

Now that you understand the individual commands, let's see how they come together in real-world workflows. These patterns represent how professional developers actually use Git day-to-day.

Basic Feature Development Flow

This is the most common workflow you'll use as a developer. It follows the pattern: sync with main ‚Üí create branch ‚Üí develop ‚Üí push ‚Üí open PR ‚Üí merge ‚Üí clean up. This keeps your work isolated until it's ready to merge. 

# 1. Update main branch
git checkout main
git pull origin main

# 2. Create feature branch
git checkout -b feature/new-feature

# 3. Make changes and commit
git add .
git commit -m "Implement new feature"

# 4. Push to remote
git push origin feature/new-feature

# 5. Open PR on GitHub
# 6. After approval, merge via GitHub UI
# 7. Clean up
git checkout main
git pull origin main
git branch -d feature/new-feature
Syncing Fork with Original Repository

When contributing to open source, you'll fork someone else's repository. This workflow keeps your fork synchronized with the original project (called "upstream"), ensuring you have the latest changes before making contributions. 

# Add original repo as upstream
git remote add upstream https://github.com/original/repo.git

# Fetch from upstream
git fetch upstream

# Merge upstream changes
git checkout main
git merge upstream/main

# Push to your fork
git push origin main

Pro Tips and Best Practices

  • Commit often: Small, frequent commits are better than large, infrequent ones ‚Äî they make it easier to track down bugs and understand the evolution of your code
  • Write meaningful commit messages: Your future self (and teammates) will thank you. Use present tense and be specific about what changed and why
  • Never commit secrets: API keys, passwords, tokens, and credentials don't belong in version control. Use environment variables and .env files (and add them to .gitignore!)
  • Use .gitignore: Exclude build artifacts, dependencies, IDE files, and OS files from being tracked (we cover this in detail in the advanced blog)
  • Pull before push: Always sync with remote before pushing to avoid conflicts and ensure you have the latest changes
  • Branch for everything: Never work directly on main ‚Äî always create a feature branch, even for small changes
  • Review before commit: Use git diff to verify what you're committing. This catches accidental changes and debug code
  • Learn to resolve conflicts: They're inevitable in team projects. Understanding how to resolve merge conflicts is a critical skill
  • Don't force push to shared branches: git push --force rewrites history and can cause major issues for collaborators
  • Keep commits atomic: Each commit should represent one logical change. If you can't describe it in one sentence, it's probably too big
Understanding Merge Conflicts

A merge conflict occurs when Git can't automatically merge changes because two people modified the same part of a file differently. This is normal and happens frequently in collaborative projects.

When conflicts happen:

  • You and a teammate both edited the same line in a file
  • Someone deleted a file while you modified it
  • Multiple branches made incompatible changes to the same code

How to resolve conflicts:

# When you encounter a conflict during merge/pull:
# 1. Git marks conflicted files in your working directory
git status  # Shows which files have conflicts

# 2. Open the conflicted file(s) in your editor
# You'll see conflict markers:
# <<<<<<< HEAD
# Your changes
# =======
# Their changes
# >>>>>>> branch-name

# 3. Manually edit the file to resolve the conflict
# Remove the markers and keep the code you want

# 4. Mark as resolved and commit
git add conflicted-file.py
git commit -m "Resolve merge conflict in conflicted-file.py"
Conflict resolution tip: When in doubt, communicate with your team! Sometimes the right solution requires understanding the intent behind both changes. Don't just blindly accept yours or theirs ‚Äî think about what makes sense for the project.
