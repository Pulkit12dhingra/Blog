1. The Two Reward Signal Architectures

There are fundamentally two ways to structure reward signals during agent trajectories:

Outcome Reward Model (ORM): Reward Only at the End

An ORM evaluates the final output and gives a single reward (or score) for the entire trajectory. For example:

  • Code task: Run all tests at the end. If they pass, reward = 1. If any fail, reward = 0.
  • Math task: Check the final answer. If it matches, reward = 1. Otherwise reward = 0.
  • Long agent task: Look at the final result. Did the agent achieve the goal? Reward = 1 or 0.

Advantages of ORM:

  • Simple to implement: Just evaluate the end state.
  • No intermediate annotation needed: You don't need to decide what's "good" at step 5 or 15.
  • Computationally cheap: One evaluation per trajectory, not many.

Disadvantages of ORM:

  • Sparse feedback: In a 50-step trajectory, you get exactly one reward signal. The agent doesn't know which of the 50 steps contributed to success or failure.
  • Credit assignment nightmare: If step 5 was a bad decision but step 40 recovered, how does the model learn? It doesn't - the final outcome was good, so all steps get equal credit.
  • Scaling issue: As task complexity grows, the probability of randomly getting the right sequence becomes vanishingly small. The agent struggles to find winning trajectories at all.
Process Reward Model (PRM): Reward at Each Step

A PRM evaluates the trajectory at each step and provides feedback throughout. For example:

  • Code development: After each function definition, syntax check, or test run, provide feedback on whether that step improved things.
  • Math reasoning: After each line of work, verify the algebraic steps are correct. Reward = 1 if correct, 0 if not.
  • Long agent task: After each action, check whether the agent is making progress toward the goal.

Advantages of PRM:

  • Dense feedback: In a 50-step trajectory, you get up to 50 reward signals. The model knows which steps are good and which are bad immediately.
  • Credit assignment clarity: If step 5 was bad, the model sees immediate negative feedback and learns to avoid that action pattern.
  • Exploration efficiency: The model learns faster because it gets more gradient signal per episode.
  • Long-horizon capability: Models can tackle much longer tasks because each step is evaluated in isolation.

Disadvantages of PRM:

  • Requires step-level annotations: You need to define what makes a "good" intermediate state. For some tasks, this is easy (test passes). For others, it's hard.
  • Computationally expensive: You run the evaluator at every step, not just the end. A 50-step task costs 50x more evaluation than ORM.
  • Annotation overhead: For tasks where you can't auto-verify each step, you need human raters at each step. This is prohibitively expensive.

2. Concrete Example: Debugging a Python Script

Let's say an agent is tasked with fixing a broken Python script. It can take up to 20 actions: read the error, hypothesize a bug, modify the code, run tests, iterate, etc.

With ORM (Outcome-only):
  • Action 1: Agent reads the error message → no reward yet
  • Action 2: Agent adds a print statement → no reward yet
  • Action 3: Agent runs the script → no reward yet
  • ... (17 more actions)
  • Action 20: Agent runs final tests → NOW we evaluate. Tests pass? Reward = 1. Tests fail? Reward = 0.

Problem: if tests fail, the agent has no idea which of the 20 actions was wrong. Was it action 3? Action 15? Was the final modification unnecessary and bad? The model must figure this out entirely from the one binary signal at the end. This is extremely sample-inefficient.

With PRM (Step-level):
  • Action 1: Agent reads the error message → Verify: is the error message correctly parsed? Reward = 1 or 0.
  • Action 2: Agent adds a print statement → Verify: does this actually help diagnose the issue? Reward = 1 or 0.
  • Action 3: Agent runs the script → Verify: does the modified code execute? Reward = 1 or 0.
  • ... (17 more actions, each evaluated immediately)
  • Action 20: Tests pass? Reward = 1. Tests fail? Reward = 0.

Now the agent gets step-level feedback. If action 15 was a bad code change, the agent sees immediately that it failed the step-level verifier. It learns to avoid that pattern. This is much more efficient.

3. Why PRM Dominates for Agentic Tasks

The research on this is clear: for multi-step reasoning and agent tasks, PRMs significantly outperform ORMs. Here's why:

  • Mathematical reasoning: In a 10-step proof, if step 7 is wrong, a PRM catches it immediately. An ORM says "your final answer is wrong" with no guidance on where the error originated.
  • Code generation: A syntax error on line 5 compounds through the entire generated solution. A PRM detects it at line 5. An ORM only sees the failure at the end.
  • Planning tasks: A bad routing decision early on wastes many subsequent steps. A PRM penalizes it early. An ORM credits the agent for trying, even though the plan is doomed.

The tradeoff is real: PRM requires defining intermediate evaluation criteria, and it's more computationally expensive. But for agents that must reliably solve multi-step problems, it's non-negotiable.

4. When Can We Use ORM? When Must We Use PRM?

Task Characteristics ORM Viable? PRM Needed?
Single decision (translate, summarize) ✓ Yes No
Few steps (3-5), clear goal ✓ Maybe Helps, but ORM works
Multi-step (10-20), complex reasoning ✗ No ✓ Strongly recommended
Long horizon (40-100 steps) ✗ No way ✓ Essential
Intermediate states hard to evaluate ✓ Acceptable Difficult to implement

5. Complete Code: ORM vs PRM Comparison

The simulation below mirrors a real multi-step agent episode and shows exactly where each reward architecture injects learning signal.

  • Episode generator (`build_episode`): Creates a 20-step trajectory where each step has a quality score, representing how useful each action was.
  • Reward wiring: Assigns ORM reward only at the terminal step and PRM reward at every step, so the only difference is signal timing.
  • Return computation (`discounted_return`): Converts raw rewards into training targets used by policy updates.
  • Comparison routine (`compare_orm_vs_prm`): Measures how many steps actually carry signal and quantifies density ratio.
import numpy as np
import pandas as pd

# Build a synthetic multi-step trajectory
def build_episode(steps: int = 20, seed: int = 11) -> pd.DataFrame:
    """Generate a trajectory with step quality scores."""
    rng = np.random.default_rng(seed)
    
    # Each step has some inherent quality
    quality = rng.uniform(0.2, 1.0, size=steps)
    
    frame = pd.DataFrame({
        "step": np.arange(1, steps + 1),
        "action_quality": quality.round(3)
    })
    
    # ORM: Only give reward at the final step
    frame["orm_reward"] = 0.0
    frame.loc[steps - 1, "orm_reward"] = 1.0  # Success at end only
    
    # PRM: Give reward proportional to step quality
    frame["prm_reward"] = frame["action_quality"]
    
    return frame

# Compute discounted returns for learning signal
def discounted_return(rewards: np.ndarray, gamma: float = 0.95) -> np.ndarray:
    """Compute cumulative discounted returns (used in policy optimization)."""
    returns = np.zeros_like(rewards, dtype=float)
    running = 0.0
    
    # Backward pass: compute cumulative future rewards
    for i in range(len(rewards) - 1, -1, -1):
        running = rewards[i] + gamma * running
        returns[i] = running
    
    return returns

# Compare the two approaches
def compare_orm_vs_prm():
    # Generate episode
    episode = build_episode(steps=20, seed=11)
    
    # Compute returns for both
    orm_returns = discounted_return(episode["orm_reward"].to_numpy())
    prm_returns = discounted_return(episode["prm_reward"].to_numpy())
    
    episode["orm_returns"] = orm_returns
    episode["prm_returns"] = prm_returns
    
    # Compute signal statistics
    orm_nonzero = float((episode["orm_reward"] > 0).sum())
    prm_nonzero = float((episode["prm_reward"] > 0).sum())
    
    stats = {
        "orm_steps_with_signal": orm_nonzero,
        "prm_steps_with_signal": prm_nonzero,
        "signal_density_ratio": prm_nonzero / max(orm_nonzero, 1),
        "orm_average_return": episode["orm_returns"].mean(),
        "prm_average_return": episode["prm_returns"].mean(),
    }
    
    return episode, stats

# Run the comparison
episode_data, statistics = compare_orm_vs_prm()

print("=== Trajectory Data (first 5 steps shown) ===")
print(episode_data[["step", "action_quality", "orm_reward", "prm_reward", "orm_returns", "prm_returns"]].head())

print("\n=== Signal Density Analysis ===")
for key, value in statistics.items():
    if isinstance(value, float):
        print(f"{key}: {value:.2f}")

print("\n=== Key Insight ===")
print(f"ORM provides signal to {int(statistics['orm_steps_with_signal'])} step (final only)")
print(f"PRM provides signal to all {int(statistics['prm_steps_with_signal'])} steps")
print(f"Learning signal density is {statistics['signal_density_ratio']:.0f}x higher with PRM!")

# Visualize the difference
print("\n=== ORM vs PRM: Learning Signal Comparison ===")
print("Step | Action Quality | ORM Return | PRM Return")
print("-" * 50)
for idx, row in episode_data[::4].iterrows():  # Show every 4th step
    print(f"{int(row['step']):3d}  |      {row['action_quality']:.2f}       |    {row['orm_returns']:.3f}    |   {row['prm_returns']:.3f}")

When you run this code, you'll see that ORM provides a single reward of 1.0 at the final step, creating a sparse learning signal. PRM, by contrast, provides dense feedback throughout the trajectory - approximately proportional to the quality of each step. This density difference is why PRM dramatically outperforms ORM on long-horizon tasks.

6. Building a Verifier for Each Step

If you commit to PRM, you need step-level verifiers. Here are examples:

  • Code writing: After each function/block, run syntax check and unit tests. Score = number of passing tests / total tests.
  • Math: After each algebraic step, check whether the equation is valid (use symbolic computation or expert verification).
  • Planning: After each action, check whether the agent is closer to the goal (distance to goal decreases? Resource constraints still met?).
  • Web navigation: After each click/action, check whether the agent is closer to the target (did page load correctly? Did navigation succeed?).

The key insight: you need domain-specific verifiers for each task type. But the payoff is enormous: your agents learn much faster and solve much harder problems.

7. Real-World Data: ORM vs PRM Performance

In OpenAI's o1 and DeepSeek-R1 research, models trained with process rewards (step-level feedback) dramatically outperform models trained with only outcome rewards (final answer only):

  • Mathematical problem solving: PRM-trained agents solve ~20% more problems correctly.
  • Code generation: PRM-trained agents pass ~15% more test suites with fewer attempts.
  • Multi-step reasoning: PRM-trained agents show better generalization to out-of-distribution reasoning patterns.

The reason: PRMs provide much richer learning signal, reducing the sample complexity of learning long-horizon policies.

8. Summary and Implications

For short, well-defined tasks: ORM is sufficient and cheaper.

For long-horizon agent tasks: PRM is nearly mandatory. The dense feedback enables much more reliable learning.

For deployed agents in the wild: The best-performing systems use PRM selectively - evaluating steps that matter most and skipping steps where feedback is obvious.
