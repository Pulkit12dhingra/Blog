The Problem With Sequential Thinking

In Part 3 we built a pipeline where agents run one after the other. That's fine when Step B genuinely needs Step A's output. But what if three agents are completely independent of each other? Running them sequentially means waiting for Agent 1 to finish before Agent 2 even starts — and Agent 3 waits for both. That's wasted time.

Imagine asking your AI system for a market report. It needs to research the news, check stock data, and scan social media sentiment — three tasks that have nothing to do with each other. Running them sequentially takes 3× as long as running them in parallel.

The ParallelAgent solves this. It launches all its sub-agents at the same time and waits for all of them to finish, then collects their outputs.

When to use ParallelAgent:
  • Multiple independent research or data-gathering tasks
  • Running the same task on different inputs simultaneously (fan-out)
  • Generating multiple variations of something at once
  • Calling multiple external APIs concurrently
  • Any workflow where sub-tasks do not depend on each other's results

How ParallelAgent Works

When a ParallelAgent receives a user message, it fires off all its sub-agents simultaneously. Each agent runs in its own execution branch with a copy of the conversation history at the point the parallel section started. When all agents have completed, their outputs are merged back into the main conversation history — in the order the agents were listed in sub_agents.

Because each sub-agent gets its own execution branch, they cannot directly read each other's real-time output mid-run. However, they all write to the shared tool_context.state dictionary, so after all agents complete, any downstream agent (e.g., an aggregator in a subsequent SequentialAgent step) can read all their results from state. 


# Execution timeline comparison

# Sequential (3 agents × 2 seconds each = 6 seconds total):
t=0s  agent_1 starts
t=2s  agent_1 done  →  agent_2 starts
t=4s  agent_2 done  →  agent_3 starts
t=6s  agent_3 done

# Parallel (all 3 agents × 2 seconds each = 2 seconds total):
t=0s  agent_1, agent_2, agent_3 all start simultaneously
t=2s  all three done

Step 1: Tools for Each Research Agent

Each research agent has a dedicated tool for its job and a dedicated state-saving tool so its results can be retrieved by the aggregator.

import os
from dotenv import load_dotenv
from google.adk.agents import Agent, ParallelAgent, SequentialAgent
from google.adk.tools import ToolContext

load_dotenv()

# ── News tools ────────────────────────────────────────────────────────

def fetch_news(company: str) -> dict:
    """Fetch recent news headlines for a company.

    Args:
        company: The company name to search for.

    Returns:
        Dict with a list of recent news headlines.
    """
    # Replace with a real news API (e.g., NewsAPI, SerpAPI) in production
    return {
        "company": company,
        "headlines": [
            f"{company} reports record Q1 revenue, beating analyst expectations.",
            f"{company} announces new AI-powered product line at annual conference.",
            f"Analysts upgrade {company} stock to 'buy' following strong earnings.",
        ]
    }


def save_news(summary: str, tool_context: ToolContext) -> dict:
    """Save news summary to shared session state.

    Args:
        summary: Summarised news findings.
        tool_context: ADK tool context.

    Returns:
        Confirmation dict.
    """
    tool_context.state["news_summary"] = summary
    return {"status": "news saved"}

Step 2: Build the Three Parallel Research Agents

Each agent is given exactly one job. Their instructions tell them to call their fetch tool, distil the result, and save it to state. They do not produce a long final answer — their job is data collection, not synthesis. 


# ── Parallel research agents ──────────────────────────────────────────

news_agent = Agent(
    name="news_agent",
    model="gemini-2.0-flash",
    description="Fetches and summarises recent news for a company.",
    instruction=(
        "You are a news research specialist. When given a company name, call fetch_news "
        "to retrieve recent headlines. Write a 2-3 sentence summary of the key news themes. "
        "Then call save_news with your summary. Keep it factual and concise."
    ),
    tools=[fetch_news, save_news],
)

financials_agent = Agent(
    name="financials_agent",
    model="gemini-2.0-flash",
    description="Fetches and summarises key financial metrics for a company.",
    instruction=(
        "You are a financial data analyst. When given a company name, call fetch_financials "
        "to retrieve key metrics. Write a 2-3 sentence summary highlighting the most important "
        "financial signals (growth, margins, valuation). Then call save_financials with your summary."
    ),
    tools=[fetch_financials, save_financials],
)

sentiment_agent = Agent(
    name="sentiment_agent",
    model="gemini-2.0-flash",
    description="Fetches and summarises social media sentiment for a company.",
    instruction=(
        "You are a social media analyst. When given a company name, call fetch_sentiment "
        "to retrieve sentiment data. Write a 2-3 sentence summary of the sentiment landscape "
        "and key trending topics. Then call save_sentiment with your summary."

Step 3: Build the Aggregator Agent

This agent runs after all parallel agents complete. It reads the three state summaries and produces a final unified report. 


# ── Aggregator agent ──────────────────────────────────────────────────

aggregator_agent = Agent(
    name="aggregator_agent",
    model="gemini-2.0-flash",
    description="Combines news, financial, and sentiment research into a unified market report.",
    instruction=(
        "You are a senior market analyst. Call load_all_research to retrieve the summaries "
        "gathered by the research team. Synthesise all three into a cohesive 200-250 word "
        "market intelligence report. Structure it with three short sections: "
        "News & Events, Financial Health, and Market Sentiment. "
        "End with a 1-sentence overall outlook. Write clearly for a non-technical executive audience."
    ),
    tools=[load_all_research],
)

Step 4: Compose ParallelAgent + SequentialAgent

Here's the key architectural pattern: we wrap the three research agents in a ParallelAgent, then wrap that parallel block together with the aggregator in a SequentialAgent. This gives us: parallel research first, then sequential synthesis. 


# ── ParallelAgent — all three researchers run at the same time ────────

research_team = ParallelAgent(
    name="research_team",
    description=(
        "Runs news, financials, and sentiment research concurrently. "
        "All three agents execute in parallel and save results to shared state."
    ),
    sub_agents=[news_agent, financials_agent, sentiment_agent],
)


# ── SequentialAgent — research first, then aggregate ─────────────────

market_report_pipeline = SequentialAgent(
    name="market_report_pipeline",
    description=(
        "Full market intelligence pipeline: parallel research phase followed by "
        "a sequential aggregation phase. Returns a unified market report."
    ),
    sub_agents=[research_team, aggregator_agent],
)

root_agent = market_report_pipeline

Key Concepts Reference

Concept Detail
Execution All sub-agents start simultaneously. ParallelAgent waits for all to finish before returning.
Context isolation Each sub-agent gets its own execution branch — they cannot read each other's real-time outputs mid-run.
State sharing tool_context.state is shared across all agents. Use it to pass results from parallel agents to downstream agents.
Output ordering When outputs are merged back into history, they appear in the order agents were listed in sub_agents, regardless of completion order.
No LLM routing ParallelAgent is a workflow agent — it does not use Gemini to decide what to run. All sub-agents always run.
Combining with SequentialAgent The most powerful pattern: put ParallelAgent inside a SequentialAgent to do parallel work then sequential synthesis.

Combining All Three Workflow Agents

In real systems, you will often combine all three workflow agents into more complex pipelines. Here is a typical layered pattern:


# Real-world pattern: validate → enrich in parallel → review in a loop → finalise

pipeline = SequentialAgent(
    name="full_pipeline",
    sub_agents=[
        validator_agent,                 # Step 1: validate input
        ParallelAgent(                   # Step 2: enrich from multiple sources at once
            name="enrichment",
            sub_agents=[source_a, source_b, source_c],
        ),
        LoopAgent(                       # Step 3: iteratively improve until quality passes
            name="quality_loop",
            sub_agents=[drafter, reviewer],
            max_iterations=3,
        ),
        publisher_agent,                 # Step 4: publish the final result
    ],
)
ADK Composition Rules:
  • Any workflow agent can be nested inside another as a sub-agent.
  • A plain Agent can sit alongside workflow agents in the same sub_agents list.
  • SequentialAgent + ParallelAgent is the most common pattern: do parallel work, then sequential synthesis.
  • LoopAgent is typically placed inside a SequentialAgent as one step of a larger pipeline.

When NOT to Use ParallelAgent

  • When agents depend on each other. If Agent B needs Agent A's output, use SequentialAgent — not ParallelAgent.
  • When you have a single-step task. No benefit to parallelism if there's only one thing to do.
  • When state writes conflict. If two parallel agents write to the same state key, the last write wins — an unpredictable race condition. Use unique state keys per agent.
  • When you need strict ordering of results. While output ordering in history is deterministic, execution timing is not. Don't rely on one parallel agent finishing before another.

Final Thoughts

The ParallelAgent is your go-to tool whenever you have multiple independent tasks that can run simultaneously. Combined with SequentialAgent for chaining and LoopAgent for iteration, you have the full ADK workflow toolbox to build sophisticated, production-grade AI pipelines.

The key insight from this series: choose your agent type based on your control flow needs, not your data needs. Use SequentialAgent for ordered dependence, LoopAgent for iteration until done, and ParallelAgent for concurrent independence. Layer them freely to model any real-world workflow.
